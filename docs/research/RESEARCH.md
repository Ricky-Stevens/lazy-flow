## Part 1 — Architect Synthesis (decisive)


### 1.1 Executive Summary & Thesis

## Thesis

**lazy-flow is a self-hostable, open-source software-delivery intelligence platform that matches Pluralsight Flow on the deterministic metric catalog and beats it 2x on insight by owning the three things Flow structurally cannot do: (1) explainable, evidence-cited AI judgment instead of black-box scores, (2) a unified GitHub+Jira value-stream view instead of two silos, and (3) radical transparency — every formula published, every AI verdict auditable, shipped as a Claude Code plugin where the LLM that narrates the data lives in the same surface the engineer already trusts.**

## What "best-in-class" means here

The research converges on a clear definition. Best-in-class is NOT more metrics — it is **correct, defensible, gaming-resistant, and ethically-bounded** metrics. Three primitives separate credible products from naive dashboards, and we must nail all three:

1. **Correct Jira changelog parsing.** Every flow metric (cycle time, time-in-status, flow efficiency, WIP, CFD, aging) reduces to one correct pass over the changelog. Three traps sink most implementations: (a) the initial status is NOT in the changelog (seed from `fields.created` + `fields.status`); (b) history is NOT ordered (sort by `created`); (c) status→category mapping is absent (fetch `/status` separately, key off numeric IDs not localized strings). Getting all three right is a correctness moat.

2. **HALOC, not raw LOC, as the universal code-change unit.** `per_hunk = max(insertions, deletions)`, summed. This kills git's modify-line double-counting and underpins every volume/churn/work-type metric. Cheap, deterministic, language-agnostic.

3. **Per-workflow Flow State Model.** The same status ("In UAT") is active for one team, waiting for another. A global mapping is wrong for most teams. This config — AI-seeded, human-confirmed, confidence-scored — is where flow-efficiency accuracy lives or dies.

## How we exceed Pluralsight Flow

Flow's headline metric (Impact) is its weakest point: GitClear shows commit signals correlate only ~27% with actual effort, and the weighting is undisclosed, so engineers cannot trust or reproduce it. We win by **replacing every black box with a transparent deterministic core PLUS an explainable LLM judgment layer that cites its evidence**:

- **Flow under-serves flow-efficiency** (it's Jira-only and can't see the code phase). We fuse GitHub commit-activity (active) and PR review round-trips (wait) into a true value-stream efficiency, and detect *implicit* wait states (assignee gaps, blocked flags, no-activity windows) that naive implementations miss.
- **Flow's Investment Profile needs manual ticket labeling.** We classify work-type from the diff via LLM with a deterministic conventional-commit/work-type-recency prior, and fall back to the git-blame work-type split when issue links are missing — so it produces a credible number even at low linkage.
- **AI-code-quality is a near-empty competitive lane.** GitClear researches churn/copy-paste/AI-authored %; nobody ships it well as a live product metric. This is our defensible 2026 wedge.
- **We turn perception/system divergence and anomaly explanation into features** — Flow ships raw charts; we ship cited, contestable explanations.

## Non-negotiables baked into the product

- **No individual leaderboards, ever.** All comparison/ranking is team-or-higher aggregate. Per-person data is private self-view only. This is both the dominant ethical line in the research (SPACE/DORA/DevEx all explicit) and a marketing differentiator versus surveillance-flavored incumbents.
- **Every metric carries a trust tier** (deterministic / hybrid / probabilistic) and an in-product "how is this computed?" link.
- **The MCP server owns all credentials and the shared DB**; clients never hold a broad PAT or DSN.

### 1.3 Deterministic vs Probabilistic Ruling

## The ruling

**Default to deterministic. Use AI only where a number genuinely cannot be counted or diffed — and even then, never let the LLM produce the number directly.**

The research is unanimous on the winning pattern: **every "probabilistic" metric is actually hybrid**. Deterministic features are computed first and handed to the LLM as *evidence to reason over*, never as arithmetic to perform. The LLM's job is rubric-bounded judgment and attribution with cited evidence — it never invents a magnitude it could hallucinate.

### DETERMINISTIC (pure computation — the bulk of the catalog)
All timing, counting, and structural metrics: Deployment Frequency, Lead Time, Recovery Time, all PR phase/latency metrics, Time-in-Status, Cycle/Flow Time, WIP, Throughput, Aging WIP, CFD, HALOC, work-type split, Rework%, Nagappan-Ball, cyclomatic/cognitive complexity, nesting, LOC counts, Monte Carlo forecasting (simulation over historical data is deterministic given a seed), velocity/say-do (with start snapshot), reviewer-load distribution.

**Rule:** if it's a timestamp subtraction, a count, an AST walk, a blame-age comparison, or a statistical simulation, it is deterministic. No LLM. Publish the formula.

### HYBRID (deterministic core + LLM judgment layer)
Change Failure Rate (det. deploy count + LLM/heuristic incident-causation linkage), Deployment Rework Rate (det. deploys + classify planned/unplanned), Flow Efficiency (det. durations + AI-seeded human-confirmed active/wait classification), Flow Distribution & Work Classification (det. conventional-commit/path prior + LLM on diff), Investment Balance, Ticket-Work Alignment, Effort Proportionality, Velocity Anomaly Explanation, PR Quality Score, explainable Impact, AI-code-quality, Estimation-miss *reasons*.

**Rule for hybrids — five mandatory constraints (from the LLM-as-judge research):**
1. **Deterministic features first**, passed as evidence; LLM never computes magnitude.
2. **Pointwise rubric scoring** (avoids position bias, which swings 0.23–0.82), **low-precision scales** (binary or 0–4, never 1–10), **temperature 0**, **constrained decoding** (schema-grammar) so output is valid by construction.
3. **Every claim cites evidence** (quoted diff hunk / signal value) or returns "insufficient information." For generative attribution (anomaly explanation), constrain to a **closed menu** of candidate causes — no free-form narrative.
4. **Effort/ordering, not absolute numbers** — LLMs preserve relative ordering far better than magnitude; effort is always a ratio/ordinal band.
5. **Calibrate against a human-labelled gold set** (Cohen's κ ≥ 0.6, macro-F1 for classification, Spearman for effort) and validate self-reported confidence against actual accuracy.

### PROBABILISTIC-LEANING (LLM-dominant, advisory only)
Review Depth/substantiveness, communication sentiment, the *narrative* in velocity explanations. Ship aggregate-only, advisory, never scored into a KPI, never per-individual.

### What we deliberately do NOT fake
DXI and all perceptual/satisfaction metrics are **survey-only by design**. Any system-derived "DXI" is a fabrication. We either build an open, published-formula survey module (turning Flow/DX's black-box criticism into our advantage) or omit it and say so — but we never dress deterministic system data up as a perceptual score.

### Rationale
Engineers distrust un-auditable numbers; the entire competitive thesis rests on transparency. A wrong-but-confident Flow Efficiency dashboard is worse than none. The deterministic/hybrid/probabilistic tier is therefore not just an implementation note — it is **surfaced in the UI as a trust badge on every metric**.


### 1.4 Competitive Edge (how we beat Pluralsight Flow)

## The five wedges that make lazy-flow 2x Flow on insight

**1. Explainable Impact replaces the black box (attacks Flow's headline at its weakest point).**
Flow's Impact correlates ~27% with real effort and has undisclosed weighting — its flagship is its credibility hole. We ship a transparent deterministic core (HALOC, edit-locations, files, change-entropy, old-code%) PLUS an LLM-judged complexity/risk layer that returns a **rationale string per commit** ("touched auth middleware + a migration; high blast radius"). Every factor and weight is visible and configurable. We win the one metric Flow leads on.

**2. Unified GitHub+Jira value stream (Flow's two silos become one).**
Flow's flow-efficiency is Jira-only and blind to the code phase. We fuse commit-activity (active) and PR review round-trips (wait) into a true end-to-end efficiency, and **detect implicit wait states** (assignee gaps, "changes requested" idle, blocked-flag durations, no-activity windows) that even the Flow Framework's own authors call the central unsolved problem. We merge LinearB's 4-phase cycle time *and* Haystack's First-Response/Rework/Idle review decomposition into one view — competitors make you pick one tool.

**3. AI-code-quality as a first-class category (a near-empty lane).**
GitClear researches churn-within-2-weeks, copy-paste rate, moved-lines %, and AI-authored % across 153M lines; no competitor ships these as live product metrics. Our positioning for 2026: "AI is shipping more code — here's whether it's *durable* code." This is the single most defensible wedge for a GitHub-sourced tool, and it requires the LLM access we already have in the Claude Code surface.

**4. Cited, contestable AI explanations (Flow ships charts; we ship reasoning).**
A red velocity line becomes a ranked, evidence-pointed hypothesis ("review latency tripled + WIP doubled in week 32"). Per-criterion ticket-work alignment with diff quotes. We surface perception/system divergence as a feature. Every AI verdict is auditable (prompt version, model, evidence) and **correctable by the engineer** — corrections become calibration signal. Flow cannot follow without an LLM in the loop.

**5. Transparency + the Claude Code surface as the moat.**
Every formula published; every metric tagged with its trust tier; engine/version pinning documented (exactly how we count `&&`/`||`/switch). Shipped as a Claude Code plugin, the LLM that narrates the deterministic numbers lives where the engineer already works — and a bundled subagent only *narrates* numbers the MCP server computed exactly, so insight is conversational without sacrificing accuracy.

## Plus the table-stakes we must match exactly
HALOC unit, 30-day (configurable) work-type split, Efficiency=100−Rework%, full PR/review suite, regex ticket-linking, Cycle Time from changelog — formula-identical so migration off Flow is trivial. We then improve each: configurable churn window (Flow hard-codes 30d), business-hours-aware latency (Flow appears not to), LLM-augmented ticket linking for commits with no issue key (Flow silently under-links), substantive-vs-nit review classification (Flow's "Thoroughly Reviewed" degrades to any-comment).


### 1.5 System Architecture (research recommendation)

## System architecture

**Shape: a remote MCP server (Streamable HTTP) that owns all credentials, ingestion, storage, and metric computation; the Claude Code plugin is a thin transparent client.** This is the single most important architectural decision — it gives one shared ingestion path, one shared DB, server-held secrets, and auto-reconnect, instead of N drifting per-developer local stdio servers.

### Layers

**1. Ingestion (server-side)**
- **GitHub: GitHub App** (not PAT) — org-admin installs once, fine-grained least-privilege scopes, rate limits scale to 12,500/hr per installation, built-in webhooks, short-lived (1hr) installation tokens. Token-per-installation pool, per-installation rate-limit tracking, pause at safety margin.
- **Jira: single distributable OAuth 2.0 (3LO) app** — read-only scopes only (`read:jira-work`, `read:board-scope`, `read:sprint`, `offline_access`). Never per-user API tokens (violates Atlassian policy).
- **Three-phase sync (both sources):** (a) **backfill once** — paginate history; GitHub via `since`/cursor checkpoints, Jira via `/search/jql` cursor + `bulkfetch` changelog (NEVER inline `expand=changelog` — it truncates at 100 histories); (b) **real-time webhooks** as latency accelerators (at-least-once, lossy, so triggers-not-truth); (c) **reconciliation sweep** — ETag-conditional GitHub polls (304s are free) + Jira `updated >= lastSync` JQL with overlap window, the authoritative completeness backstop. Jira dynamic webhooks expire in 30 days → cron refresh.
- **Hybrid REST+GraphQL on GitHub:** REST for bulk single-type discovery (commits via `since`, deployments), GraphQL to hydrate the rich per-PR object graph (reviews+comments+timeline+commits) in one call, killing the N+1. Embed `rateLimit{cost,remaining}` and stay under 2,000 points/min secondary limit and 500k-node ceiling.
- **Idempotency:** dedupe webhooks on `X-GitHub-Delivery`; upsert keyed on `node_id` / Jira `issue.id`, last-writer-wins gated by `updated_at`, so out-of-order webhook+poll deliveries converge.

**2. Storage** (see Data Model section)

**3. Deterministic Metric Engine (server-side)**
- Pure SQL/code over the normalized core + materialized `metric_snapshots`. Computes all Group A–E metrics on-the-fly for any date range AND writes immutable daily rollups for fast dashboards and historical freezing.
- The **Jira changelog parser** is the keystone: seed initial status, sort histories, map status→category via separate fetch, key off numeric IDs.
- A **code-analysis worker** runs `web-tree-sitter` (WASM grammars, no native build — critical for plugin distribution) on base/head SHA per PR for complexity + per-function deltas; `git blame` for work-type/churn; `scc` for LOC only.

**4. AI Engine (server-side)**
- Builds the deterministic feature/evidence pack, calls Claude with pointwise rubric prompts, temperature 0, constrained decoding (schema grammar). Persists full audit record. Includes the calibration harness against gold sets. Gates ensemble (majority vote) on low confidence.

**5. MCP / Plugin surface**
- **One plugin (`lazy-flow`)** via a git-hosted marketplace: skills (slash commands like `/lazy-flow:dora`, `/lazy-flow:cycle-time`), a narrating subagent, and one entry in `.mcp.json` pointing at the remote HTTP server.
- MCP server exposes deterministic metrics via `registerTool` with `outputSchema` + `structuredContent` so values are exact and machine-checkable; dashboards/snapshots as MCP **Resources** for at-mention.
- A bundled subagent produces probabilistic narratives **only from the deterministic tool outputs** — the model narrates, never computes.
- Team-wide install: commit `extraKnownMarketplaces` + `enabledPlugins` to repo `.claude/settings.json`; org enforcement via managed settings + `strictKnownMarketplaces` allowlisting only our repo.

### Sync model summary
Backfill → webhook-accelerated incremental → ETag/JQL reconciliation, all idempotent, with a surfaced **Sync Freshness / watermark-lag** health metric so users know if a dashboard is current before acting.


### 1.6 Data Model & Storage (research recommendation)

## Recommended store

**Postgres for multi-team installs; SQLite as the zero-config default for single-team self-hosters; DuckDB attached for heavy ad-hoc analytics.** Keep the schema SQL-portable (TEXT-for-JSON on SQLite, JSONB on Postgres via a dialect shim). SQLite/DuckDB are single-writer/single-process and cannot serve concurrent multi-user dashboards — so Postgres is the answer the moment more than one team queries concurrently. DuckDB (columnar, 20–50x faster on aggregations) attaches to the same files for large-date-range scans without adding a server.

This mirrors the proven Four Keys pattern: raw events in, derive via scheduled SQL.

## Two-layer schema

**Layer 1 — system-of-record (faithful mirror, raw payload retained).** Every row keeps its full upstream JSON in a `raw` column so metric definitions can change without re-fetching from rate-limited APIs — a decisive advantage over tools that discard raw data; re-derivation is just re-running SQL.

Core tables:
- `organisations`, `repositories(github_node_id UNIQUE, default_branch)`, `jira_projects`
- `commits(sha PK, author_identity_id, authored_at, committed_at, additions, deletions, raw)`
- `pull_requests(id PK, author_identity_id, created_at, merged_at, merged_by_identity_id, first_commit_at, first_review_at, approved_at, state, raw)` — **stage timestamps denormalized** so cycle-time queries don't re-walk events
- `reviews(pr_id, reviewer_identity_id, state, submitted_at)`, `review_comments(pr_id, author_identity_id, created_at, raw)`
- `deployments(repo_id, sha, environment, status, created_at, finished_at, raw)`
- `issues(id PK, project_id, key, type, status_id, status_category, story_points, assignee_identity_id, created_at, resolved_at, raw)`
- `issue_transitions(issue_id, from_status_id, to_status_id, transitioned_at, actor_identity_id)` — **append-only, rebuilt from changelog, sorted by timestamp on ingest** (this is the keystone for all flow metrics)
- `sprints(id, state, start_at, end_at, complete_at)`, `sprint_issues(sprint_id, issue_id, points_at_start, added_after_start)` — **membership snapshotted at sprint start** since Jira keeps no history

**Layer 2 — derived/operational:**
- `sync_state(source, resource, scope_id, cursor, last_run_at, status, error)` — per-resource high-water marks
- `metric_snapshots(scope_type, scope_id, metric, day, value, window, trust_tier)` — immutable daily rollups, scope_type team-or-higher only
- `ai_verdicts(subject_id, metric, prompt_version, model_id, temperature, feature_vector, structured_verdict, evidence, confidence, created_at)` — full audit trail per AI verdict
- `flow_state_models(workflow_id, status_id, flow_state, confidence, confirmed_by)` — the per-workflow active/wait/new/done map

## Identity stitching (foundational — wrong stitches produce defamatory stats)

Three-tier, deterministic-first, reversible:
- `persons(id PK, display_name, primary_email_hash)` — canonical human
- `identities(id PK, person_id FK NULLABLE, kind ENUM('github_login','commit_email','jira_account'), external_id, confidence, raw)`
- Match: verified-email exact (conf 1.0, auto-merge) → normalized email-local-part + name (0.8) → fuzzy name (0.5, queued for human confirmation). Handle GitHub `noreply` emails as un-matchable. `person_id` nullable so unmerges are trivial; never hard-delete; every merge audited.

## Rationale & guardrails
- **Raw retention** enables retroactive metric revision (most competitors can't).
- **Snapshot freezing + on-the-fly recompute** gives stable historical numbers yet auditable definitions.
- **No first-class per-person productivity snapshots** — scope_type is team-or-higher by design; person-level data is private self-view computed on demand, never persisted as a ranking.
- Emails/accountIds are PII (GDPR): hash at rest, support per-person deletion, per-tenant isolation (never cross-client benchmarking).
- Migrations: plain checked-in up/down SQL, in-DB version table, forward-only in prod, one set with a thin dialect layer for SQLite+Postgres.
- API tokens are secrets: OS keychain / server env, never in the DB or repo.


### 1.7 AI Insight Designs

All four insights follow the same skeleton: **deterministic feature pack → pointwise rubric prompt (temp 0, constrained decoding) → schema-valid structured verdict with per-claim evidence → audit record → calibration against gold set.** Specifics below.

### 1. Ticket-Work Alignment Score
- **Inputs (deterministic):** Jira description + parsed acceptance criteria, issue type/summary; PR title/body, commit messages; relevance-ranked diff hunks (never silent truncation — rank hunks against criteria); the deterministic Jira↔PR link.
- **Prompt strategy:** pointwise. For each parsed criterion c_i emit `{covered: yes/no/unclear, evidence: <quoted diff hunk>}`. **A criterion may only be "covered" if a diff quote is supplied; no quote ⇒ "unclear."** Plus a 0–4 ordinal band (0 unrelated → 4 fully addresses + verifiable test coverage).
- **Structured output:** `{ordinal: 0-4, criteria: [{text, covered, evidence}], coverage_ratio, confidence}`. Final = min-rule of ordinal band and coverage ratio.
- **Calibration:** human-labelled gold set, target κ ≥ 0.6.
- **Confidence:** lowered when criteria are too vague to assess (surface "criteria too vague," don't guess).
- **Auditability:** persist prompt version, model, evidence quotes. **Use:** flags silent scope drift and under-specified tickets — process insight, never an individual cudgel.

### 2. Effort Proportionality
- **Inputs (deterministic):** effort vector {HALOC, files touched, #commits, cycle time, review rounds, #comments, rework commits}; ticket scope text; team historical effort distribution; story points as one signal (not ground truth).
- **Prompt strategy:** classify into ordinal `{much_lower, lower, as_expected, higher, much_higher}` vs expected, with few-shot scale-aware exemplars of similar past tickets. **Output ordinal + log-ratio, NEVER a raw point prediction** (LLMs preserve ordering, not magnitude).
- **Structured output:** `{band, log_ratio, expected_basis, confidence}`. Cross-checked against deterministic cycle-time z-score; disagreement lowers confidence.
- **Guards:** exempt research/spike issue types (structurally unestimable); descriptive only (Goodhart — lines/commits are gameable); never per-developer.
- **Use:** surfaces under-scoped tickets that ballooned (planning risk) or trivial changes that took weeks (blockers).

### 3. Velocity Anomaly Explanation
- **Detection (deterministic):** control-chart/z-score or EWMA on the throughput/cycle-time series; flag |z|>2 with a minimum sample size (resist small-team noise/seasonality).
- **Signal pack (deterministic):** WIP, reviewer latency, #blocked tickets, ticket churn (re-opens/AC edits), PTO/holidays, team-size delta, large-PR share, incident/hotfix volume, dependency wait.
- **Prompt strategy:** rank a **CLOSED MENU** of candidate causes by likelihood; each requires an **evidence pointer** to a signal-pack value; model MUST select "insufficient signal" rather than invent. No free-form attribution.
- **Structured output:** `[{cause: enum, confidence, evidence_pointer}]`, phrased "consistent with," never "caused by."
- **Ethics:** attribute to systemic signals (review bottleneck, scope churn), never name individuals.
- **Use:** turns a red dashboard line into a verifiable hypothesis in seconds — the key differentiator over Flow's raw charts.

### 4. Velocity Baselining (per-person private + company-wide)
- **Deterministic at core:** per-person baseline = own throughput/cycle-time trend vs own history (private self-view only, no percentile). Company-wide = aggregate distribution across teams. AI layer only *narrates* the deterministic trend ("your cycle time rose ~30% this month, concentrated in review wait").
- **Guards:** private self-view enforced by access control; org view is aggregate-only; never a cross-person ranking.

### Cross-cutting AI engineering rules
- Pointwise over pairwise (position bias swings 0.23–0.82). Binary or 0–4 scales, never 1–10 (judges cluster on 6–7). Temperature 0. Constrained decoding (Anthropic structured outputs — schema compiled to grammar, guaranteed valid JSON, no parse-retry; note numeric min/max + recursive schemas unsupported, handle refusals/cutoffs).
- **Ensemble (majority vote across model families) gated on low confidence only** — resolves >95% of residual bias but doubles cost.
- **Calibration harness is a first-class feature:** gold set per team, report κ / macro-F1 / Spearman, iterate rubric until agreement closes, validate self-reported confidence against actual accuracy.
- **Full audit record persisted per verdict** (prompt version, model+snapshot, temperature, feature vector shown, structured verdict, evidence, confidence) — this auditability is the product moat.
- For any employee-performance framing, route through qualified HR/legal review.


### 1.8 Ethical Guardrails

## Concrete, enforced-in-code guardrails

These are not policy docs — they are architectural constraints that make abuse structurally impossible.

### 1. Team-vs-individual (the hard line)
- **No per-individual leaderboards, percentiles, rankings, or forced curves — ever.** The product literally will not generate them. SPACE/DORA/DevEx authors are all explicit; Microsoft and Amazon both retreated from stack ranking after it destroyed collaboration.
- **All comparison/ranking is team-or-higher aggregate.** It is technically impossible to drill from a team aggregate down to an individual ranking.
- **Per-developer data is private self-view only:** visible to that developer alone, compared only against their *own* history (no cross-developer percentile), framed for growth, enforced by access control. This aligns with GDPR subject-access while preventing surveillance.
- **An "ethics mode" is ON by default:** blocks export of individual data into review/comp/layoff workflows and warns when usage drifts toward evaluation.

### 2. Transparency
- Every metric shows its **formula inline** ("how is this computed?") and its **trust tier** (deterministic / hybrid / probabilistic) as a badge. No black-box scores.
- Engine/version pinning is published (exactly how `&&`/`||`/`switch` are counted, which churn window, which DORA report version for benchmarks).
- Every AI verdict is **auditable** (prompt version, model, evidence pointers, confidence) and **contestable** — engineers can correct it, and corrections feed calibration.

### 3. Anti-gaming (Goodhart defense)
- **Default to balanced multi-metric views** (SPACE-style: pair speed with a quality/experience counter-metric). No single composite "productivity number." Optimizing many metrics at once is hard, raising gaming cost.
- **Active gaming detection, not just computation:** flag deployment-frequency inflation (non-prod/rapid redeploys), CFR suppression (deploys with later hotfixes but no incident), lead-time resets from squash/rebase, status-juggling that inflates flow efficiency, trivial-PR-splitting. Report the number PLUS a data-quality/confidence indicator.
- **Never ship raw LOC or commit-count as productivity** anywhere. HALOC and effort-weighting kill the LOC-padding incentive.
- Surface DORA's own Goodhart warning in-product when a user tries to set a metric as a goal.
- Effort/PR-size are **descriptive context, flagged not hard-penalized** (large refactors are legitimate).

### 4. Privacy & compliance
- Per-developer performance data is **personal data (GDPR).** Systematic profiling triggers an **Art. 35 DPIA** — ship a built-in DPIA template + legitimate-interest assessment generator and a transparency-notice generator. **Employee consent is not a valid legal basis** (power imbalance) — guide buyers to documented legitimate interest.
- Data minimization, purpose limitation, defined retention limits, per-tenant isolation (treat all customer data as client-owned; never benchmark one client against another).
- Hash emails/accountIds at rest; support per-person deletion (subject erasure).

### 5. Bias & altitude
- AI metrics are advisory, aggregate-only, with a human in the loop; never used for individual evaluation.
- Don't auto-flag a Flow Distribution or work-mix as "bad" — no universally correct mix; humans set targets with business context.
- Keep rubrics about substance, not eloquence/verbosity (avoid Anglophone-prose bias in PR quality scoring).

### Caveat
This is product/ethics scaffolding, not legal advice — buyers must route DPIA and legitimate-interest assessments through qualified data-protection counsel.

### 2.1 DORA / Four Keys metrics

DORA (DevOps Research and Assessment, now part of Google Cloud) defines a small set of software delivery performance metrics validated across the multi-year *Accelerate State of DevOps* research program. The classic "Four Keys" are split into **throughput** (Deployment Frequency, Change Lead Time) and **stability** (Change Failure Rate, Failed Deployment Recovery Time / Time to Restore Service). The 2024/2025 framework evolved this: stability was renamed **instability**, Failed Deployment Recovery Time was reclassified under throughput, and a fifth metric, **Deployment Rework Rate**, was added. **Reliability** is treated as an operational performance outcome (availability, latency, SLO attainment) layered on top of the four keys rather than as a fifth core delivery metric — it has no single canonical formula and is the least standardized.

The open-source **Google Four Keys** project is the reference implementation. It ingests webhook events (GitHub deployments, commits/pushes, issues; Cloud Build; GitLab pipelines) into BigQuery `events_raw`, then derives three tables — `deployments`, `changes`, `incidents` — keyed on commit SHA and deployment_id. Concretely: a *deployment* is a `github` `event_type=deployment` event (or a successful Cloud Build / GitLab pipeline); a *change* is a commit (SHA + author timestamp); an *incident* is a GitHub issue tagged with an `incident` label whose body references a deployment. **Lead Time** = median(deployment_time − commit_authored_time) joined on SHA. **Deployment Frequency** is bucketed (daily/weekly/monthly) rather than a raw rate. **Change Failure Rate** = deployments-with-linked-incident ÷ total deployments. **Time to Restore** = median(incident_resolved − incident_created).

For a GitHub+Jira-only product, the cleanest mapping is: Deployment Frequency and Lead Time from the GitHub Deployments/Releases API + PR merge/commit data; Change Failure Rate and Recovery Time from **Jira incident tickets** (issue type = Incident, or a label), linked to deployments by time-window or by smart-commit/PR references. Most inputs are **deterministic** given clean linkage; the weak points — identifying which deployments *caused* a failure, and classifying a deployment as planned vs. rework — are **hybrid** (deterministic if teams tag incidents/link tickets, otherwise need heuristic or LLM inference). The dominant gaming risks are deployment-frequency inflation (splitting deploys, counting non-prod), CFR suppression (not filing incident tickets, mislabeling), and lead-time distortion (rebasing/squashing resets commit timestamps). DORA explicitly warns these are *team* health signals, not individual performance metrics.

**Recommendations**

## What this product should adopt

**1. Implement the Four Keys algorithm faithfully, keyed on SHA + deployment_id.** Build three internal entities — `deployments`, `changes`, `incidents` — exactly as Google's Four Keys does, but source deployments/changes from the GitHub API and incidents from Jira. This is the only methodology with a published, validated reference implementation; mirroring it makes our numbers defensible against "you computed DORA wrong" objections.

**2. Make the deployment definition explicit and configurable.** Default to the GitHub Deployments API with `deployment_status=success`, with Releases and a named deploy workflow_run as alternatives. Let teams choose, and always filter to production. This is the single biggest source of cross-vendor disagreement — owning it transparently is a differentiator.

**3. Use median, not mean, for Lead Time and Recovery Time; bucket Deployment Frequency.** Match DORA exactly. Show distributions (p50/p75/p90), not just a single number — Pluralsight Flow's weakness is opaque single-number scores.

**4. Lean on Jira for the stability metrics, and link deterministically.** Identify incidents by Jira issue type = Incident (or a configurable label), pull resolution time from the **changelog status transition** (not the closed date), and link incident→deployment via (a) smart-commit/PR references first, (b) release/version field second, (c) time-window heuristic last. Surface the linkage method per data point so users see confidence.

**5. Classify each metric's trust level in the UI.** Deterministic (Deploy Freq, Recovery Time) vs. hybrid (Lead Time, CFR, Rework, Reliability). For hybrid metrics that need inference (e.g., planned-vs-rework, incident causation), use an LLM pass over PR/ticket text but **show the inferred label and let users correct it** — corrections become training signal and build trust.

## What to avoid / do better than competitors

- **Do not present Reliability as a computed core metric** from GitHub+Jira alone — it needs APM telemetry. Offer it as a clearly-caveated proxy (incident rate/severity) or an integration hook. Overclaiming here will be the most obvious credibility hole.
- **Never attribute DORA metrics to individuals.** DORA, SPACE, and DevEx authors are explicit that these are team/system metrics; individualizing them drives the exact gaming behaviors below. This is also an ethical/anti-surveillance line our org policy reinforces.
- **Actively detect gaming, don't just compute.** Flag deployment-frequency inflation (deploys to non-prod, rapid redeploys), CFR suppression (deploys with zero linked incidents but later hotfix commits), and lead-time resets from squash/rebase. Competitors report the number; we should report the number plus a data-quality/confidence indicator.
- **Pin to a named DORA report version for benchmarks** (e.g., 2024 four-cluster vs. 2025 percentile bands) and show it — the cluster definitions shift yearly and the 2024 medium-below-high CFR anomaly shows static thresholds mislead.

### 2.2 Flow Framework (Kersten / Tasktop)

The Flow Framework (Dr. Mik Kersten, "Project to Product", 2018; commercialized by Tasktop, now Planview Viz) reframes software delivery as a value stream producing business value, measured end-to-end from "yes, let's do this" (business approval) to "end-user obtains value" (production + validation). Its unit is the Flow Item — anything a stakeholder pulls through the stream — in exactly four mutually-exclusive types: Features (new value/revenue generation), Defects (quality/revenue protection), Risks (security, compliance, governance), Debt (architectural/technical debt removal). Every item moves through four abstract Flow States: New → Active → Waiting → Done. This 4-type x 4-state model is the entire computational substrate.

The five Flow Metrics are deterministic aggregations over items and state durations: Flow Velocity (throughput: items reaching Done per period), Flow Time (calendar-elapsed entry→Done, including weekends/wait/off-hours — explicitly NOT cycle time, NOT lead time), Flow Efficiency (sum active time / Flow Time), Flow Load (WIP: items in Active+Waiting), Flow Distribution (percentage mix of the four types). These are value-stream metrics meant to be "approximately right rather than exactly wrong," analyzed as trends; Flow Time supports percentile predictability (e.g. P85 <= 30 days).

Flow Efficiency is the marquee, hard metric. It is computed entirely from a Jira changelog (status history): each status the item traverses is mapped to active/waiting/new/done via a per-workflow Flow State Model; sum active durations and divide by total Flow Time. The framework's deepest insight and biggest data trap is status-semantics ambiguity — the same Jira status ("In UAT") is active in one team's workflow but waiting in another's, so classification must be per-workflow, not global. Industry Flow Efficiency is shockingly low: typically 5-15%, with >15% considered "good" and 30-40% near-ideal — most delivery time is pure waiting (queues, blocks, handoffs), exactly the waste the metric exposes.

For a GitHub-only signal the framework maps less cleanly: PR open→merge is a cycle-time slice not Flow Time, but GitHub gives excellent active/wait decomposition for the code phase (commit activity = active; "waiting for review" / "changes requested" idle = wait). Labels/PR type infer Flow Distribution. The framework is deliberately tool-agnostic and abstraction-heavy; both the value and the risk live in the state-mapping configuration — human judgment, therefore gameable and inconsistent.

## What THIS product should adopt, avoid, and beat

**Adopt (table stakes):**
- **Implement Flow Efficiency from the Jira changelog as the headline differentiator.** Pluralsight Flow and most GitHub-centric tools under-serve this. The algorithm is deterministic given a state map: pull `status` changelog histories, build timestamped intervals, map each status to {new,active,waiting,done}, efficiency = sum(active) / FlowTime. This is the wedge: most competitors only do PR cycle time (a sub-segment), not value-stream active/wait decomposition.
- **Build a per-workflow Flow State Model as a first-class, editable config** — never a global default. Seed it with an LLM from status names + workflow structure + observed dwell patterns, then REQUIRE human confirmation and show a confidence score. Directly addresses the "In UAT" ambiguity the framework's own authors call the central problem.
- **Count items, not story points**, for Velocity/Distribution (points are gameable, non-comparable). Report Flow Time as median + P85/P95, never mean (outliers dominate).
- **Time-weight aggregate Flow Efficiency** (sum-active/sum-flowTime), not mean of per-item ratios — the latter lets tiny items dominate.

**Beat competitors by:**
- **Detecting implicit wait states** workflows lack explicitly: assignee gaps, "changes requested" idle on PRs, blocked-flag durations, no-activity windows. Solves the structural over-reporting that plagues naive implementations and the framework itself (workflows without Waiting statuses report falsely high efficiency).
- **Fusing GitHub into the value stream**, not as a separate silo: commit activity to active, PR-ready-but-unreviewed and review round-trips to wait, giving a true code-phase Flow Efficiency that Jira-only Tasktop can't see. Use Conventional Commit prefixes + PR labels to auto-seed Flow Distribution.
- **Little's Law forecasting** (FlowTime ~= FlowLoad/Velocity) as a forward-looking date predictor and data-quality cross-check — flag when measured Flow Time and WIP/throughput disagree (signals state-mapping errors).

**Avoid (gaming/ethics):**
- **Never expose Flow Efficiency or Velocity as individual-performance scores.** They measure SYSTEM waste; ranking people guarantees status-gaming (teams stop using wait statuses, efficiency reads 90%+, data dies). Keep altitude at team/value-stream; gate per-person views.
- **Don't auto-flag a Flow Distribution as "bad"** — no correct mix; it's strategy-dependent. Surface it, let humans set targets.
- **Warn loudly on low-confidence state maps and missing wait states** rather than emitting a confident-but-false number — a wrong Flow Efficiency dashboard is worse than none.
- Treat all customer Jira/GitHub data as client-owned; never benchmark one client's streams against another's without explicit consent.

### 2.4 DevEx / DX Core 4

**Two related-but-distinct frameworks.** (1) **DevEx (Noda, Storey, Forsgren, Greiler — ACM Queue / CACM, 2023, "DevEx: What Actually Drives Productivity")** is a *human-centric* model built on three dimensions — **Feedback Loops** (speed/quality of responses to a developer's actions), **Cognitive Load** (mental effort to do a task), and **Flow State** (uninterrupted, immersive focus). These three roll up 25+ sociotechnical factors. The paper's central methodological claim is that DevEx must be measured with **both perceptual data (surveys: attitudes/feelings) and workflow/system data (objective instrumentation)**, plus a third technique, **experience sampling** (short in-the-moment prompts). A core insight: perception and system data often disagree (fast review turnaround can still *feel* disruptive), and the perceptual half is irreducible — it cannot be derived from GitHub/Jira alone.

(2) **DX Core 4 (Abi Noda / DX, late 2024)** is the *operational* framework that unifies DORA + SPACE + DevEx into **four dimensions — Speed, Effectiveness, Quality, Impact** — each with one headline metric and several secondaries, deliberately balanced as counter-metrics (Speed is paired with DXI so you can't sacrifice experience for throughput). The canonical headline metrics are **Diffs/PRs per Engineer** (Speed), **Developer Experience Index / DXI** (Effectiveness), **Change Failure Rate** (Quality), and **% time on new capabilities** + **Revenue per Engineer** (Impact). DX prescribes capturing each dimension three ways: **self-reported survey, system metric, and experience sampling**, explicitly acknowledging some signals are only perceptual.

**The DXI** is DX's proprietary headline: a composite index (commonly cited as **14 actionable drivers**, instrumented via a longer ~40-item Likert survey — "DX Snapshots"), grounded in research across 800+ orgs / 4M+ data points. Named drivers include deep work / focus time, local iteration speed, release process, code review speed, code quality/maintainability, documentation, build & test, cross-team collaboration, and confidence in making changes — i.e. it operationalizes feedback loops + cognitive load + flow state into one number. DX claims **every +1 DXI point ≈ 13 min/developer/week saved (~10 hrs/year; ~1,100 hrs/yr per 100 engineers)**. DXI is **survey-based by design** and cannot be computed from GitHub/Jira.

**System-measurable vs perceptual (key for a GitHub+Jira-only product):** Deterministically computable from GitHub/Jira are throughput (PRs/diffs), lead/cycle time, deployment frequency, time-to-merge, time-to-first-review, CFR (from incident/revert linkage), MTTR, time-to-10th-PR, interruption proxies (context-switch counts). **Not computable without surveys:** DXI, perceived ease of delivery, perceived rate of delivery, perceived software quality, cognitive load felt, flow/focus satisfaction. Several Impact metrics (innovation ratio, defect ratio) are **hybrid** — DX derives them by LLM-classifying PRs. **Implication for our product:** we can authentically deliver the *workflow half* of DevEx and ~half of DX Core 4 deterministically, approximate the AI-classified Impact/Quality splits probabilistically, but must be explicit that DXI/perceptual dimensions require a survey layer we either build or omit (and must not fake from system data).

## What THIS product (GitHub + Jira, Claude Code plugin) should do

**1. Adopt DX Core 4 as the top-level scaffold, but be honest about the perceptual gap.** Speed, Effectiveness, Quality, Impact is the right org-recognized structure and unifies DORA/SPACE/DevEx. Ship the *deterministic* metrics we can genuinely compute from GitHub/Jira: Diffs/PRs per Engineer, lead time, deployment frequency, time-to-first-review, time-to-merge, CI feedback latency, CFR, MTTR, time-to-10th-PR. These are our credible, defensible core.

**2. Deliver the *workflow half* of DevEx authentically; clearly label the perceptual half as out-of-scope or survey-gated.** We can build strong system-side proxies for all three DevEx dimensions — feedback-loop latency (review/CI times), cognitive-load proxies (review iterations, churn, files-per-PR, onboarding time), and flow proxies (context-switch/interruption counts). **Never present these as the full DevEx score.** The paper is explicit: perception ≠ system data and they routinely diverge. Our differentiator vs Flow can be *transparent, open-formula* workflow metrics rather than a black box.

**3. Do NOT fake DXI.** DXI is survey-only and proprietary. Two honest options: (a) omit it and say so, or (b) build an **open DevEx survey module** (Likert items mapped to feedback loops / cognitive load / flow) and publish our aggregation formula — turning Flow/DX's biggest criticism (opaque black box, vendor lock-in) into our advantage. If we offer a composite index, give it our own name and a documented formula; never label a system-derived number "DXI."

**4. Treat AI-classified metrics (innovation ratio, defect ratio) as probabilistic, with confidence surfaced.** Use an LLM to classify PRs/Jira issues into new-capability vs maintenance vs bug, but expose classification confidence and let users correct labels. Prefer existing Jira issue types/labels first (deterministic), LLM only as fallback — this both improves accuracy and reduces gaming.

**5. Hard ethical guardrails — bake into the product, not just docs.**
- Block all per-individual rankings on throughput metrics (PRs/diffs per engineer). Aggregate to team/org only. This is the single biggest abuse vector and a stated red line in every source.
- Always render Speed metrics paired with a quality/experience counter-metric so throughput can't be optimized in isolation (DX's core design principle).
- Exclude or external-flag Revenue per Engineer (layoff-incentivizing, not GitHub/Jira-sourceable).
- Flag bot/automation activity and exclude from human throughput.

**6. Make perception/system divergence a feature.** Where we have both a survey item and its workflow counterpart (e.g. perceived ease of delivery vs measured lead time), surface the gap — the DevEx paper frames this divergence as the most actionable insight. No competitor leans into this well.

### 2.5 Pluralsight Flow metric catalog

Pluralsight Flow (formerly GitPrime, acquired 2019, rebranded 2020; now an Appfire product) is a Git + ticket analytics platform whose catalog splits into three families: **Coding/commit metrics** (from git diffs), **PR/Review metrics** (from GitHub PRs + comments), and **Ticket/Delivery metrics** (from Jira/ADO). Two foundational primitives underpin everything.

**HALOC (Hunk-Aware Lines of Code)** is the universal unit of code change. For each diff hunk, Flow counts insertions and deletions and takes the *larger* of the two as that hunk's HALOC, then sums across hunks. This deliberately avoids the double-counting that raw `git diff` produces when a line is modified (which shows as 1 delete + 1 insert = 2). HALOC feeds Rework, Efficiency, Help Others, Legacy Refactor, tt100, and more.

**Impact** is the flagship proprietary metric — a model-derived score answering "roughly how much cognitive load did the engineer carry?" Per GitPrime's canonical docs it combines ~5-6 factors: amount of code, % of edits to *old* code, surface area (number of distinct edit locations), number of files affected, severity of edits to old code, and how the change compares to the project's history. It is summed per commit, then per author; "Weekly Impact" = sum(Impact)/sum(active weeks), where weeks with zero commits are excluded.

**Work-type breakdown** uses a strict 30-day ownership window on each changed line: **New Work** (net-new code not replacing anything), **Legacy Refactor** (edits to code >30 days old), **Help Others** (edits to *another* author's code <30 days old), **Churn/Rework** (author rewrites/deletes *their own* code <30 days old). **Efficiency** = 100 × (1 − Rework_HALOC / Total_HALOC); GitPrime's data science put typical churn at 13-30% (70-87% Efficiency).

**Activity metrics**: Coding Days (distinct days with ≥1 commit/week), Commits per Day (total commits / coding days), tt100 Raw (hours to produce 100 raw LOC) and tt100 Productive (hours per 100 LOC after removing rework).

**Review/Submit metrics**: Review Coverage (% of PR hunks commented on), Responsiveness (submitter latency to respond to comments via comment or commit), Reaction Time (reviewer latency to first response), Receptiveness (ratio of follow-on commits to comments, expected 10-20%), Influence/Reviewer Influence (did a reviewer's comment elicit a follow-on commit; all prior discussants get credit; Enterprise Server only), Involvement (% of PRs a reviewer participated in; inverse of Unreviewed PRs), Time to First Comment (hours from PR open to first comment), Time to Merge (formerly Time to Resolve/Close), Comments per PR, Thoroughly Reviewed PRs (≥1 substantive non-submitter comment / merged PRs; benchmark ~80%). A PR counts as "reviewed" if merged with ≥1 comment (GitHub approvals count as comments).

**Delivery/Investment**: Cycle Time (ticket Active→Done), and the Investment Profile report which buckets ticket effort (by ticket count, story points, or dev-weeks) into business categories. Ticket-to-commit linking is regex-based on issue keys in commit messages/branches/PR titles (default `[A-Z]+-\d+`).

**Recommendations**

## What to adopt, beat, and avoid vs Pluralsight Flow

### Adopt (table stakes — match these to be credible)
- **A HALOC-equivalent normalized code-change unit.** Implement `per_hunk = max(insertions, deletions)` from the GitHub diff as the base unit for all volume metrics. This is the single most important, cheap, deterministic primitive and avoids git's modify-line double counting. We can compute it directly from GitHub PR/commit diffs.
- **The 30-day work-type split** (New Work / Legacy Refactor / Help Others / Rework) via `git blame` line-age + author attribution. Fully deterministic from GitHub. **Improve on Flow by making the 30-day window configurable per repo** (mono-repos and slow-moving services want different thresholds) — Flow hard-codes 30 days.
- **Efficiency = 100×(1 − Rework/Total HALOC)** and **Cycle Time = Active→Done from Jira changelog.** Both deterministic, both directly reproducible. These are the metrics buyers will A/B against Flow, so match the formula exactly.
- **The full PR/review suite** (Review Coverage, Time to First Comment, Time to Merge, Responsiveness, Reaction Time, Involvement, Unreviewed PRs). All deterministic from GitHub PR + comment timestamps. Easy parity wins.
- **Regex ticket-to-commit linking** (`[A-Z]+-\d+` in messages/branches/PR titles) as the deterministic baseline for the Investment Profile and Cycle Time.

### Beat (where Flow is weak — our differentiation, leaning on the Claude Code plugin's LLM access)
- **Replace the opaque Impact black-box with an explainable, hybrid Impact.** Flow's biggest credibility gap (per GitClear: ~27% effort correlation, undisclosed weighting) is that no one can reproduce or trust it. Ship a **transparent deterministic core (HALOC, files, edit locations, old-code %) PLUS an LLM-judged complexity/risk layer** that returns a rationale string per commit ('touched auth middleware + migration; high blast radius'). Show the factors and the reasoning. This directly exceeds Flow on the one metric that is its headline and its weakest point.
- **Semantic work-type and Investment classification.** Flow's Investment Profile depends on manual ticket labeling. Use the LLM to classify each linked ticket/PR into Feature / Tech-debt / KTLO / Bug / Security from title+body+diff — eliminating Flow's labeling-hygiene dependency. Mark this probabilistic and always show the evidence.
- **LLM-augmented ticket linking** for commits with no issue key (Flow silently under-links these). Heuristic + LLM match by branch/PR text and diff content; flag as inferred (probabilistic) vs explicit (deterministic).
- **Quality-aware review depth.** Flow's 'Thoroughly Reviewed' degrades to any-comment. Use the LLM to classify comments as substantive vs nit/LGTM, giving a genuine review-quality score instead of comment counts.
- **Business-hours / timezone-aware timing** for all PR latency metrics. Flow does not appear to apply a working-hours calendar — a quick, defensible accuracy win.

### Avoid (Flow's anti-patterns and ethical traps)
- **Do not surface individual Commits-per-Day, tt100, or raw Impact as performance-ranking metrics.** These are the classic Goodhart/gaming targets (split commits, pad files, spread trivial edits). Present them only as team-level trends and outlier-detection signals, with explicit anti-stack-ranking guardrails in the UI — and document the gaming vector for each.
- **Never present a single opaque score as ground truth.** Every probabilistic/LLM metric must ship with its evidence and a deterministic fallback, so leaders can audit it. This is both an ethics requirement and our competitive wedge against Flow's black box.
- **Don't treat high New Work / low Churn as universally 'good.'** Bake phase-context and healthy-iteration caveats into the narrative the plugin emits.

### Net positioning
Match Flow's deterministic catalog exactly (so migration is trivial), then win on **explainability + LLM-derived semantic layers** (Impact rationale, ticket/PR classification, review-quality, inferred linking) that Flow structurally cannot offer without an LLM in the loop — while being louder and clearer than Flow about gaming risks and individual-measurement ethics.

### 2.6 Competitor metric catalogs

Nine SEI/DORA competitors cluster into four metric philosophies, each with distinctive ideas worth stealing.

**Flow/cycle-time decomposers (LinearB, Haystack, Swarmia).** LinearB's signature is the four-phase Cycle Time breakdown (Coding → Pickup → Review → Deploy), each a deterministic timestamp subtraction from GitHub PR events. It layers PR-quality metrics that are genuinely novel: PR Maturity (how much a PR changes after first opening), Review Depth (comments/PR), PRs Merged Without Review, and a New/Refactor/Rework work-type split derived from how recently each changed line was last touched. Haystack subdivides review into First Response Time, Rework Time (first comment → last commit), and Idle Completion Time (rework done → merge) — a sharper diagnostic of where review latency actually lives. Swarmia adds WIP working-agreements (e.g. ≤10 open PRs/team) and Batch Size as first-class, target-able metrics rather than passive charts.

**Code-effort scorers (GitClear, Code Climate Velocity, Waydev).** All three reject raw LOC. GitClear's Diff Delta is the most rigorous: a multiplicative per-edit score DD = Σ φ·⊖·⧉·β·τ·σ filtering generated/whitespace/noise, deduplicating across branches, and applying a time-survival premium so churned code earns less. Their AI research (153M+ lines) operationalizes copy-paste rate, churn rate, and moved-lines % as AI-quality red flags — directly relevant to an AI-era product. Code Climate's Rework (% of lines rewritten within 3 weeks; elite <4%) and Productive Impact (Impact − Rework) are clean, defensible churn metrics. Waydev splits throughput into New / Churn / Legacy-Refactor / Help-Others and defines Efficiency = productive code ÷ total.

**Business-alignment platforms (Jellyfish, Faros AI).** Jellyfish's Allocations / R&D capitalization maps engineering effort (tickets + commits + reviews + payroll/roster) onto capitalizable vs non-capitalizable work for finance/audit/tax-credit defensibility. Swarmia's Investment Balance is the lighter version (New / Improvements / Productivity / KTLO; healthy = Productivity 10-15%, KTLO <30%). Faros AI leads on enterprise AI-impact: AI-generated code %, AI augmentation by activity (coding/testing/debugging/documenting), correlated against DORA + eSAT surveys.

**Deploy-centric (Sleuth).** Pure DORA plus per-deploy batch-size breakdown, MTTD→MTTR decomposition, and deploy-gating.

Best ideas to adopt: (1) phase-decomposed cycle time as the spine; (2) work-type split (New/Refactor/Rework) computed deterministically from git blame recency — a far better quality signal than churn alone and the foundation for Investment Balance; (3) a GitClear-style noise-filtered effort score; (4) PR Maturity + review-latency decomposition; (5) AI-code-quality metrics (churn, copy-paste, AI-authored %) as a category competitors barely cover; (6) Investment/Allocation balance sourced from PR↔issue links. All are deterministic or hybrid from GitHub+Jira — no metric requires data we cannot source.

## What THIS product should adopt to be twice as good

**Spine: phase-decomposed Cycle Time (steal from LinearB + Haystack).**
Make the four-phase breakdown (Coding/Pickup/Review/Deploy) the default delivery view, but go one level deeper than LinearB by splitting Review into Haystack's First Response / Rework / Idle. Competitors make you pick LinearB *or* Haystack; ship both decompositions in one view. All deterministic from GitHub PR events.

**Differentiator #1: the work-type primitive (New / Refactor / Rework).**
Compute this deterministically from `git blame` recency on every changed line. It is the highest-leverage reusable primitive in the entire space because it feeds three products at once: (a) churn/quality signals (Code Climate's Rework), (b) Investment Balance bottoms-up (no issue links required), and (c) tech-debt visibility. **Differentiator #2: AI-code-quality, treated as a first-class category.**
This is the whitespace gap. GitClear researches it; nobody operationalizes it well as a live product metric. Ship churn-within-2-weeks, copy-paste/duplication rate, and moved-lines % out of the box, and (where IDE telemetry or commit trailers exist) AI-authored %. Keep it aggregate-only; never per-developer surveillance.

**Differentiator #3: effort-weighting via a noise-filtered score.**
Adopt a GitClear-style Diff-Delta-lite (filter generated/whitespace, dedup across branches, time-survival premium) and use it to *weight* every other metric — batch size, investment balance, throughput — instead of raw LOC. This kills the LOC-gaming critique that plagues Waydev/Velocity and makes our composites credible.

**Differentiator #4: Investment Balance from PR↔issue links + work-type fallback.**
Swarmia's version degrades badly without issue linkage; ours should fall back to the work-type split when links are missing, so it still produces a number at low linkage. Offer the Jellyfish-style capitalization view as the enterprise tier — but scaffold only, with an explicit "route through your finance/accounting professional" disclaimer (not authoritative tax advice).

**Things to AVOID (anti-patterns competitors fell into):**
- No proprietary black-box composite as a *headline* (Waydev Impact, Jellyfish opaque scores) — engineers distrust un-auditable numbers; publish every formula.
- Never expose Diff Delta, Rework, or PR Maturity as individual leaderboards — Goodhart + morale destruction. Team/aggregate only.
- Don't ship raw LOC or commit-count vanity metrics at all.
- Make WIP/Batch Size *working agreements with opt-in alerts* (Swarmia), not top-down surveillance dashboards.

**Net:** be twice as good by (1) merging the two best cycle-time decompositions, (2) owning the AI-code-quality category, (3) making work-type recency the universal primitive, and (4) effort-weighting everything so the composites are gaming-resistant and fully auditable — all sourced from GitHub + Jira with zero metric requiring data we can't reach.

### 2.7 Code complexity metrics

Code complexity metrics are the strongest deterministic signal a GitHub-only delivery intelligence product can extract, because they are computed purely from source text and are language-agnostic in spirit. Six families matter: **Cyclomatic Complexity (CC)** — McCabe 1976, `M = E − N + 2P` on the control-flow graph, in practice `M = (number of decision points) + 1` (each `if`/`else if`/`case`/`for`/`while`/`catch`/`&&`/`||`/ternary adds 1). McCabe's threshold was 10 per module. **Cognitive Complexity (CCog)** — SonarSource 2017, designed because CC over-penalizes `switch` and ignores readability. Three rules: (1) +1 base increment per flow break (`if`, `else if`, `else`, ternary, `for`, `while`, `do-while`, `catch`, `switch` [once], labelled `break`/`continue`/`goto`, each recursive call, and each *sequence* of like boolean operators); (2) an additional **nesting increment** equal to the current nesting depth for the structures that themselves break flow (`if`/loops/`catch`/ternary/`switch`), so an `if` at depth 2 costs +3; (3) no penalty for shorthand and no increment for the method itself. Boolean sequences score +1 per run of the same operator (`a && b && c` = +1; `a && b || c` = +2), with no nesting penalty. **Halstead metrics** — from operator/operand counts (n1,n2 distinct; N1,N2 total): vocabulary `n=n1+n2`, length `N=N1+N2`, Volume `V=N·log2(n)`, Difficulty `D=(n1/2)·(N2/n2)`, Effort `E=D·V`, est. bugs `B=V/3000`. Note the common GeeksforGeeks rendering of D as `(n1/2)·(N2/2)` is wrong; the correct denominator is n2. **Maintainability Index** — Oman & Hagemeister 1991: `171 − 5.2·ln(V) − 0.23·CC − 16.2·ln(LOC)`; Visual Studio normalizes to 0–100 via `max(0, MI·100/171)`; SEI variant adds a comment term. Van Deursen and others warn MI is opaque, the coefficients are un-recalibrated since 1991, and `ln(LOC)` means trivial reformatting moves the score — treat as a weak composite. **Structural counts** — nesting depth (max block depth per function), function length (NLOC/SLOC), file length, parameter count.

**Multi-language computation in a Node tool is the crux.** Five candidate engines: (1) **escomplex/typhonjs-escomplex** — only real Node lib giving CC + Halstead + MI together, but core escomplex is effectively unmaintained (last release 2015) and JS-only; typhonjs/@ponticus forks add Babel-with-all-plugins so they parse TS/JSX, but still cover only the JS family. (2) **lizard** (Python, terryyin) — battle-tested CC + NLOC + token + param counts across 16+ languages (C/C++/C#, Java, JS, TS, Python, Go, Rust, Swift, Kotlin via GDScript-style tokenizers, Ruby, PHP, Scala, Lua, Objective-C) — shells out, no Halstead/MI. (3) **tree-sitter** via `tree-sitter` (native Node bindings, fast) or `web-tree-sitter` (WASM, portable, ~slower) with per-language grammars — you write your own AST-query-based metric extractors; gives full control and uniform cross-language CC/CCog/nesting/length, at the cost of building and maintaining metric rules per grammar. (4) **SonarJS / sonar engines** — canonical CCog implementation but JVM-hosted and JS-family only in the Node-runnable form. (5) **scc/tokei** — extremely fast (Go/Rust, 150–250+ languages) for LOC/comment counts and scc's branch-counting CC *approximation*, but not true graph-based CC and no Halstead.

The git-diff angle: all metrics are file-level functions of source, so per-PR deltas come from running the engine on the file at the base SHA and the head SHA and differencing per function (matched by signature/name). This yields the most actionable signal — "this PR raised max function CCog from 12 to 41" — far better than absolute repo snapshots.

**Recommendations**

## Recommended approach for a multi-language, Node-based complexity engine

**Adopt a tree-sitter core, with lizard as a validation oracle and scc for LOC.**

1. **Primary engine: `web-tree-sitter` (WASM) with bundled grammars.** This is the only option that gives a *uniform, in-process, dependency-light* path across the languages a GitHub product must cover (TS/JS, Python, Go, Java, C#, Ruby, PHP, Rust, etc.). Ship `.wasm` grammars as assets (no native build toolchain on the user's machine — critical for a Claude Code plugin). Use native `tree-sitter` bindings only if you control the runtime and need the ~throughput.
   - Implement **Cyclomatic Complexity, Cognitive Complexity, max nesting depth, function/file length, and parameter count** yourself as tree-sitter query + small visitor per grammar. These are pure AST walks and port cleanly.
   - Compute **per-function deltas from the git diff** by analyzing the file at base SHA and head SHA and matching functions by qualified name/signature — this is the headline, actionable output and should be the product's primary surface.

2. **Use `lizard` as a cross-check, not the primary engine.** It is mature and multi-language but Python (shell-out) and gives no Halstead/MI. Run it in CI on a sample corpus to validate your tree-sitter CC numbers stay within tolerance; do not put a Python dependency on the critical path of a Node plugin.

3. **Use `scc` (or tokei) for LOC/comment counts only** — it is 10–100x faster than any JS counter and covers 250+ languages, but its CC is a branch-count approximation, so never present it as McCabe CC.

4. **escomplex / typhonjs-escomplex / @ponticus: limit to the JS/TS family if you want Halstead + MI cheaply.** Core escomplex is unmaintained (2015); prefer the @ponticus fork (Babel, TS/JSX). Do not extend it to other languages. Treat Halstead/MI as secondary trend metrics, never SLA targets.

5. **Do NOT reuse SonarJS as a runtime dependency** — it is JVM-hosted and JS-only in the embeddable form. Instead, treat the Sonar white paper as your *spec* and write conformance tests against the worked examples so your tree-sitter CCog matches the reference.

## What to do better than competitors
- **Lead with cognitive complexity deltas per PR**, not absolute repo snapshots — this maps directly to the DevEx cognitive-load dimension and is what Flow under-serves.
- **Be explicit about engine/version pinning**: publish exactly how you count `&&`/`||`, `switch`, and recursion, because cross-tool CC is otherwise meaningless.
- **Normalize by NLOC** (complexity per 100 lines) so large files don't dominate.

## What to avoid
- Never use any complexity metric — especially LOC, MI, or CC — as an individual-developer performance score. All are gameable (Goodhart) and complexity is usually inherited, not authored. Restrict to *code-health* and *team-trend* framing.
- Exclude generated/vendored/migration files or they swamp every metric.
- Do not present Halstead numbers or MI as cross-language or cross-tool comparable.

### 2.8 Code churn, Impact & rework (deterministic from git)

Measures of change significance and effort from git diffs, moving beyond raw LOC (shown by Nagappan & Ball 2005 and GitClear/Pluralsight to be a poor effort/risk proxy). Three families: (1) Churn/rework/efficiency: Pluralsight Flow Rework = author rewrites/deletes own code within 30 days; Efficiency = 100 - Rework%; healthy churn 13-30% (~25% typical, from GitPrime's 85k-engineer dataset). Work types: New Work, Legacy Refactor (>30d code edited), Help Others (another author's <30d code), Rework (own <30d). Academic root: Nagappan & Ball 8 relative measures M1-M8; M1=ChurnedLOC/TotalLOC scored Spearman 0.883 vs defect density on Windows Server 2003 (44.97M LOC), classifying fault-prone binaries at 89%, while absolute churn was weak (adj R-squared ~0.05) - the basis for relative>absolute. (2) Impact (GitPrime/Pluralsight): cognitive difficulty from insertion points (edit locations), LOC, files affected, severity (new vs edit), dispersion. (3) Diff Delta (GitClear): DD = sum over per-line events of six factors (file/branch filter, context filter zeroing whitespace/keywords/comments, duplication filter, operation base-score with move/copy/find-replace discounted, time/durability scalar, language/context scalar e.g. CSS ~40% of Python). Of 87.1M changed lines only 2.3% survive filters; LOC ~95% noise (54% formatting, 30% movement, 11% churn). HALOC (half-lines) fixes git's double-counting of modified lines (git shows 4 LOC for 2 edits; HALOC=2). All deterministic from git log --numstat/diffs except move/copy detection and refactor-vs-churn intent (hybrid/probabilistic).

**Recommendations**

**Adopt (deterministic from git log --numstat + git blame + git diff -U0):**

1. **HALOC as base unit, not raw LOC** (HALOC(hunk)=max(added,deleted)). Cheap, language-agnostic; denominator for all ratios.
2. **Self-churn / Rework% with configurable window (default 21d; expose 14/30)** via blame. Report Efficiency=100-Rework%. Always show as team trend against the 13-30% healthy band, never an individual leaderboard - the most-abused metric here.
3. **Work-type split (New/Legacy Refactor/Help Others/Rework)** - high explanatory value, reframes churn neutrally. Requires author-identity dedup built once.
4. **Nagappan-Ball M1, M2, M3 as a defect-RISK overlay** (validated rho up to 0.883, 89% classification). Surface as QA targeting, not productivity.
5. **Impact approximation** = weighted blend of edit-locations, HALOC, files-touched, change entropy, new-vs-edit severity, with transparent user-visible weights.

**Differentiate:**
- Be transparent where GitClear/GitPrime are opaque: ship every formula+weight configurable with a per-commit "why this score" factor breakdown.
- Implement the cheap 80% of Diff Delta (ignore masks, whitespace/comment/keyword stripping, copy/move/find-replace discount via git diff -M -C + block-hash dedup, durability premium). Don't claim parity - language-verbosity/operation calibration is GitClear's ~30-month moat.

**Avoid / guard rails:**
- Never expose these as individual performance rankings; churn is healthy iteration and ranking drives gaming (commit-batching, large-block dumping).
- Don't ship raw LOC as "productivity" anywhere.
- Move/rename/copy detection (-M/-C + dedup) is a hard dependency; without it refactors pollute churn and work-type splits. Label operation-classification-dependent metrics as hybrid/probabilistic.

### 2.9 Agile / Jira delivery metrics

This dimension covers flow and predictability metrics derived from an issue tracker (Jira), the bulk of which are **deterministic** given clean changelog and board data, with two genuinely **probabilistic** members (Monte Carlo forecasting and estimation-quality judgement).

**The changelog is the single most important data source.** Jira's `GET /rest/api/3/issue/{id}/changelog` returns a paginated (`startAt`/`maxResults`/`total`) list of `histories`; each history has a `created` timestamp, an `author`, and an `items[]` array where status transitions appear as `{field:"status", from, to, fromString, toString}`. Two non-obvious primary-source facts drive correct implementation: (1) **the issue's initial status and creation are NOT in the changelog** — you must seed state from the issue's `fields.created` and `fields.status`; (2) **histories are not guaranteed ordered** — you must sort by `created` yourself. `fromString`/`toString` are *localized display names*, so robust code keys off the numeric `from`/`to` status IDs and joins to the status metadata (`/rest/api/3/status` or `/statuscategory`) to map each status into the three Jira status categories: To Do / In Progress / Done. This category mapping (plus the board's column→status config from `/rest/agile/1.0/board/{id}/configuration`) is what lets you define "started" and "finished" lines for Cycle Time, Lead Time, WIP, CFD, and Aging WIP.

**Time-in-status** = sum of intervals between consecutive sorted status transitions; the duration of the final (current) status runs to "now". **Cycle Time** = time from first entry into a "started" status to first entry into a Done-category status; **Lead Time** = creation → Done. Subtract non-working time (weekends/calendar) for credible numbers — Vacanti and Atlassian both define these as elapsed-time clocks but real boards exclude blocked/queue columns by policy.

**Velocity / Say-Do / predictability** all hinge on the Sprint object (`/rest/agile/1.0/sprint/{id}` + `.../issue`) and the **Story Points custom field** (`customfield_10016`, instance-specific). Atlassian's documented subtlety: the Velocity "commitment" bar counts *all* points in the sprint at start regardless of status, while "completed" counts only points whose issues reached a resolved/Done category by sprint close — and scope added mid-sprint is excluded from commitment but included in completed. Snapshotting committed scope at sprint start (via the `sprint` field changelog or a sprint-start automation) is required because Jira does not retain it natively.

Authoritative sources: The Kanban Guide (kanbanguides.org, May 2025) and Scrum.org/ProKanban for the four flow metrics; Daniel Vacanti's *Actionable Agile Metrics for Predictability* for CFD/cycle-time-scatterplot/Monte Carlo; Atlassian support + developer docs for Jira field/API specifics.

## What this product should adopt

**1. Make the changelog parser the foundation — and get the three known traps right.** Every flow metric (cycle time, lead time, WIP, CFD, aging, time-in-status) reduces to one correct pass over the changelog. The implementation MUST: (a) seed initial state from `fields.created` + `fields.status` because the first status is absent from the changelog; (b) sort `histories` by `created` (Jira does not guarantee order); (c) key off numeric `from`/`to` status IDs (not localized `fromString`/`toString`) and join to `/rest/api/3/statuscategory` for To Do/In Progress/Done mapping. Most competitor implementations and naive scripts get at least one of these wrong — getting all three right is a correctness differentiator.

**2. Report distributions and percentiles, not averages.** Best-in-class (Vacanti/ActionableAgile) reports cycle time as 50/85/95th percentiles on a scatterplot, never a mean. Bake percentiles in as the default; means hide tail risk and erode trust.

**3. Prefer throughput + Monte Carlo over velocity for forecasting.** Story points are subjective, team-relative, and non-comparable. Offer velocity for teams that want it, but lead with estimation-free throughput-based Monte Carlo forecasts (confidence-banded dates). This is where the product can clearly beat Flow's velocity-centric framing.

**4. Snapshot sprint commitment at sprint start.** Jira does not persist start-of-sprint scope. To compute Say-Do and predictability honestly, reconstruct committed scope from the `sprint`-field changelog (or capture it at sprint-start). Document the exact definition (points vs count, mid-sprint scope handling) since Jira's own Velocity vs Sprint reports disagree.

**5. Lead with Aging WIP and the CFD as the actionable layer.** Aging WIP overlaid on cycle-time percentile bands is the only metric a team can act on mid-sprint; the CFD is the best single bottleneck diagnostic. These should be first-class, not buried.

## What to avoid

- **Do not surface per-individual velocity, cycle time, or estimation accuracy as performance scores** — all are trivially gameable (point inflation, ticket splitting, under-commitment) and ethically fraught (Goodhart's law). Frame everything as team-level process health.
- **Do not compare velocity or story points across teams** — points are not a shared unit.
- **Do not present Monte Carlo or estimation outputs as single dates** — always communicate ranges with confidence levels.
- **Classify honestly:** all core flow metrics are deterministic; only Monte Carlo forecasting (probabilistic) and the *why* behind estimation misses (hybrid/LLM) need AI. Don't dress deterministic math up as AI.

### 2.10 PR & code-review metrics (GitHub)

## PR & Code-Review Metrics: Evidence Base

The dominant industry framing for PR flow is **LinearB's Cycle Time**, which decomposes the journey from first commit to production into four sequential phases: **Coding** (first commit → PR opened), **Pickup** (PR opened/ready → first non-author review action), **Review** (first non-author review action → merge), and **Deploy** (merge → release). This phase model is the single most useful organizing structure for a Flow competitor because every phase is computable from GitHub's Timeline/Reviews/Commits APIs plus optional release/deployment data. Critically, LinearB sets pickup-end and review-start at the *first non-author* comment/approval/merge — self-reviews and author comments are explicitly ignored, and a PR merged with no review records review time = 0. Coding time requires git history (first commit timestamp on the head branch), which is the only phase needing more than the PR/Timeline APIs.

The GitHub data substrate is precise and largely deterministic. The **Timeline API** (`GET /repos/{owner}/{repo}/issues/{n}/timeline`) emits ordered events with timestamps: `review_requested`/`review_request_removed` (created_at + review_requester + requested_reviewer), `ready_for_review`/`convert_to_draft`, `reviewed` (submitted_at + state), `committed`, `head_ref_force_pushed`, `merged`, `closed`. The **Reviews API** returns review objects with `state` ∈ {APPROVED, CHANGES_REQUESTED, COMMENTED, DISMISSED, PENDING}, `submitted_at`, `user`, `body`. Two distinct comment streams exist: **review comments** (anchored to diff: path, position, side, diff_hunk) vs **issue comments** (PR-level discussion). PR objects carry `additions`, `deletions`, `changed_files`, `commits` for size; `created_at`, `merged_at`, `closed_at` for durations.

**Benchmarks from primary research.** Google's *Modern Code Review* study (Sadowski et al., ICSE-SEIP 2018, 9M reviews): 90% of changes touch <10 files, median ~24 LOC changed, >75% have a single reviewer, ~4h average turnaround (small <1h, large <5h), and reviews drive developer action ~80% of the time. SmartBear/Cisco (~2,500 reviews): optimal PR ≤200 LOC (never >400), inspection rate <300–500 LOC/hr, sessions ≤60min (never >90). Code Climate Velocity defines **Rework/churn** as LOC rewritten within 21 days (median 7%, elite ≤5%, poor >10%). DORA's 2025 **Rework Rate** is a different, deployment-based metric: unplanned bug-fix deploys ÷ total deploys. Industry SLA consensus: time-to-first-review <4h elite / median org ~15h / >24h is a bottleneck.

Most metrics are deterministic from API timestamps. The probabilistic/hybrid frontier is *qualitative*: review depth/usefulness, whether a comment is substantive vs nit, and rework-vs-normal-iteration classification — these need LLM judgment. Gaming risk is high for any individual-attributed metric (reviewer load, comments/PR, approval speed), which is why LinearB scopes phases to teams only.

**Recommendations**

## What this product should adopt

1. **Build the LinearB four-phase cycle-time model as the spine** (Coding → Pickup → Review → Deploy), but be transparent that only Pickup/Review/Time-to-Merge are pure GitHub-API metrics. Coding time needs git commit history; Deploy time needs Releases/Deployments API or Jira deploy signals. Present the stacked-bar phase breakdown — it is the single most legible delivery-speed artifact and the thing buyers expect from a Flow competitor.

2. **Adopt the strict "first non-author action" rule** for the pickup→review boundary, and rigorously filter bots (Dependabot, Copilot, CI). This one rule is what separates a credible product from a naive one — most homegrown dashboards get fooled by self-comments and bot reviews.

3. **Lead with medians/p75, never means.** PR durations are heavy-tailed; a mean-based dashboard is wrong and will be distrusted by senior engineers. Offer business-hours-adjusted variants to kill weekend/overnight inflation.

4. **Ship the deterministic core first** (pickup, review, time-to-merge, PR-size distribution, reviewers/PR, comments/PR, iterations, merge-without-review, stale detection). These are all exact from PR/Review/Timeline APIs and need no LLM — fast to build, hard to dispute.

5. **Differentiate with the hybrid layer**: LLM-classified review depth (substantive vs nit), review coverage, and rework-vs-legitimate-iteration classification. This is where pure-deterministic competitors (and homegrown scripts) cannot follow, and it directly answers the "are we rubber-stamping?" question that comment counts alone cannot.

6. **Set explicit, sourced benchmark bands** rather than vague "good/bad": PR size ≤200 LOC ideal / >400 review (SmartBear); time-to-first-review <4h elite, >24h bottleneck (Code Climate); rework median 7% / elite ≤5% (Code Climate); single-reviewer norm and ~4h turnaround (Google). Cite the source inline — credibility is the product.

7. **Add an AI-era benchmark overlay.** Classic benchmarks predate AI-assisted coding; surface the shifts (PRs +51%, review iterations threshold >2.5, merge-without-review +31%) as a distinct lens. This is a current, defensible wedge against older tools whose benchmarks are stale.

## What to avoid

- **Never attribute PR/review metrics to individuals for performance.** Follow SPACE and LinearB: team-level only. Reviewer-load distribution should be anonymized and framed strictly for load-balancing. This also aligns with the org policy against individual benchmarking and cross-client comparison.
- **Do not over-trust `updated_at`** for stale detection — compute last *meaningful* activity (commit/review/human-comment), excluding label and bot noise.
- **Exclude generated/vendored/lock files** from PR-size and rework, or every benchmark is wrong.
- **Don't conflate the two "rework" definitions** — keep DORA Rework Rate (deployment-based) and code-churn Rework (LOC-based) as separate, clearly-labeled metrics.
- **Flag gaming explicitly in-product.** Comments/PR, approval speed, and reviewer counts are all trivially gameable; show them as context, never as standalone scored KPIs.

### 2.11 GitHub API capabilities & sync strategy

## GitHub API for a delivery-analytics product

**REST vs GraphQL.** GitHub exposes two APIs over the same data. REST (api.github.com) is best for bulk listing of one entity type (commits, PRs, releases) and for endpoints with strong incremental filters. GraphQL (single /graphql endpoint) is best for deeply nested fetches — e.g. a PR with its reviews, review comments, commits, timeline items, and check runs in one round trip, eliminating the REST N+1 pattern (GitHub's own docs cite ~11 REST calls collapsing into 1 GraphQL call). The catch: GraphQL bills by a **point/node cost model**, so naive deep nesting can exhaust quota faster than REST. The winning pattern is **hybrid**: REST for incremental discovery of changed entities (it has `since` on commits and issues, and ETags that make unchanged polls free), GraphQL for hydrating the rich object graph per changed PR.

**Rate limits (exact).** REST: 5,000 req/hr authenticated (PAT/OAuth/user); 60/hr unauthenticated; GitHub App installation token starts at 5,000/hr and **scales +50/hr per repo beyond 20 and +50/hr per org user beyond 20, capped at 12,500/hr**; GitHub Enterprise Cloud raises the floor to 15,000/hr; GITHUB_TOKEN in Actions is 1,000/hr/repo. Search endpoints: 30 req/min authenticated (code search 9–10/min). GraphQL: 5,000 **points**/hr (10,000 EMU/enterprise). GraphQL cost = sum of requests needed to satisfy each connection (assuming first/last hit their max), divided by 100, rounded, minimum 1; a single call may not request **>500,000 nodes** and first/last must be 1–100. **Secondary limits** (separate, abuse-prevention): ≤100 concurrent requests; ≤900 points/min REST and ≤2,000 points/min GraphQL; ≤90 CPU-seconds per 60 real seconds; ≥1s spacing between mutating requests. Headers: `x-ratelimit-limit/remaining/used/reset/resource`; GraphQL exposes the same via the `rateLimit{cost,remaining,resetAt,nodeCount,used}` field embeddable in any query.

**Conditional requests.** Send `If-None-Match` (ETag) or `If-Modified-Since` (Last-Modified). A `304 Not Modified` **does not count against the primary rate limit** — the single most important lever for cheap high-frequency polling of list endpoints.

**Pagination.** REST uses the `Link` header (rel=next/prev/first/last), `per_page` max 100; never hand-construct page URLs. GraphQL uses cursor connections (`pageInfo{hasNextPage,endCursor}`), `first`≤100.

**Auth for team-wide install.** A **GitHub App** is the correct choice over PAT/OAuth: higher scaling rate limits (up to 12,500/hr per installation), fine-grained least-privilege permissions, org-admin-driven installation (no per-user token sprawl), webhook subscription built in, and short-lived installation tokens (1hr) minted from the App's private key.

**Webhooks vs polling.** Webhooks are push, low-latency, and don't burn rate limit — but have **no delivery guarantee** (10s response timeout = failure; no auto-redelivery; manual redelivery only within a 3-day window). So: webhooks for real-time currency, polling-with-ETags as a reconciliation backstop, and REST `since`/list for historical backfill. Architecture = backfill once (paginate history), then webhook-driven incremental + periodic ETag reconciliation sweep, all writing through idempotent upserts keyed on GitHub node_id + a delivery-id dedupe table.

**Recommendations**

## Recommended architecture for this product

**1. Authenticate as a GitHub App, not PAT/OAuth.** Org-admin installs once; you mint short-lived (1hr) installation tokens per org from the App private key. You get least-privilege fine-grained permissions, built-in webhook subscription, and rate limits that **scale to 12,500/hr** with org size — a PAT is fixed at 5,000/hr and creates token sprawl. Maintain a **token-per-installation pool** and route each org's traffic to its own bucket; track `x-ratelimit-*` per installation and pre-emptively pause at a safety margin (e.g. stop at 200 remaining).

**2. Hybrid REST+GraphQL fetch strategy.**
- REST for **discovery and bulk single-type listing**: commits (`since`/`until`/`sha` for backfill), releases, deployments, check runs.
- REST Issues endpoint (has `since`, returns PRs) or List PRs `sort=updated&direction=asc` for **change discovery** — note List PRs has no `since`, so watermark on `updated_at`.
- GraphQL for **per-PR hydration**: pull reviews + review comments + timeline items + commits + check runs in one query to avoid the REST N+1. Embed `rateLimit{cost,remaining,resetAt}` in every GraphQL call and keep per-query cost well under the 2,000 points/min secondary limit and 500,000-node ceiling; page nested connections at first:100.

**3. Three-phase sync.**
- **Backfill (once per repo):** paginate history time-bounded by commit/issue `since`; checkpoint the last cursor/timestamp per (repo, entity) so an interrupted backfill resumes idempotently. Run repos in a serial queue, not concurrently (≤100 concurrent secondary limit; prefer serial per best-practice).
- **Real-time (steady state):** subscribe to webhooks (push, push, pull_request, pull_request_review, pull_request_review_comment, check_run, deployment_status, release). Persist the raw delivery immediately, return 200 in <10s, process async.
- **Reconciliation sweep (safety net):** because webhooks have no delivery guarantee (10s timeout, no auto-redeliver, 3-day manual window), run a periodic ETag-conditional poll of list endpoints. **304s are free**, so a frequent reconciliation sweep costs almost nothing and closes gaps from dropped webhooks.

**4. Idempotency — make every write a no-op-safe upsert.**
- Dedupe inbound webhooks on `X-GitHub-Delivery` (UUID) in a processed-deliveries table before applying.
- Key all entities on GitHub's global `node_id` (stable across renames), upsert with last-writer-wins gated by `updated_at`/event timestamp so out-of-order webhook+poll deliveries converge to the same state.
- Store an ETag and Last-Modified per polled resource to drive cheap conditional re-fetches.

**5. Rate-limit governance.** Central token-bucket scheduler per installation; honor `retry-after` then `x-ratelimit-reset`; exponential backoff on secondary limits; ≥1s between any mutations; cap concurrency. Treat secondary-limit avoidance as a hard requirement — repeated violations can get the integration banned.

## What to do better than competitors (Pluralsight Flow)
- **Use timeline events for honest cycle-time** (exclude draft time via ready_for_review; handle reopened PRs) rather than naive open→merge — this is where many tools are inaccurate.
- **Layer LLM-based review-depth scoring** (probabilistic) on top of deterministic review-coverage counts to distinguish real review culture from rubber-stamping — a differentiator, but ship it **aggregate-only and advisory**.
- **Wire DORA Four Keys from Deployments + Jira incidents**, using the 2023 Failed Deployment Recovery Time definition, and be explicit about the deploy-attribution method.

## ⚠️ Ethical / gaming caveats (build these into product guardrails)
- **Never surface individual-level rankings** on volume metrics (commits, LOC, PR count, reviewer load). These are trivially gamed and create perverse incentives (Goodhart). Default all dashboards to team/aggregate; gate any per-person view behind explicit, consented use.
- LOC and raw throughput must never drive performance evaluation — position as flow diagnostics only.
- LLM-derived metrics (review depth, incident classification) are non-deterministic and biasable; label them as such and keep a human in the loop.
- Comment/PR text is sensitive content — handle per data-ownership and privacy expectations; don't cross-tenant benchmark.

### 2.12 Jira API capabilities & sync strategy

Jira Cloud's REST API v3 (`/rest/api/3`) plus the Jira Software "Agile" API (`/rest/agile/1.0`) supply everything a GitHub+Jira delivery-analytics product needs, but the data model forces you to reconstruct flow metrics from the **issue changelog** rather than read them off pre-computed fields.

**Search has just been overhauled.** The classic `GET/POST /rest/api/3/search` (offset-based `startAt`/`total`) was deprecated and shut down through 2025 (transition May–Jul 2025, progressive shutdown Aug–Oct 2025). All searching now goes through **`/rest/api/3/search/jql`** (GET or POST), which uses opaque **`nextPageToken`** cursors with an **`isLast`/absent-token** terminator and **no `total`**. To get counts you call **`/rest/api/3/search/approximate-count`** (returns an estimate). `maxResults` defaults to 50; with `fields` requested the practical cap is ~100 issues/page. A **`reconcileIssues`** parameter (accepts up to issue IDs) forces strong consistency for specific issues against Jira's eventually-consistent search index — important because freshly-updated issues can lag the index by seconds.

**Flow metrics live in the changelog.** Request `expand=changelog` on the issue/search, or use the dedicated **`GET /rest/api/3/issue/{id}/changelog`** (paginated) or **bulk `POST /rest/api/3/changelog/bulkfetch`** (up to 1000 issues, filterable by ≤10 field IDs). The inline `expand=changelog` is **capped at the 100 most recent history entries** — long-lived issues silently truncate, so for backfill you MUST use the dedicated/bulk changelog endpoints. Each `histories[]` entry has a `created` timestamp and `items[]` with `field`, `from/to`, `fromString/toString`. Gotchas (Mike Bowler): the **initial status is NOT in the changelog** (derive from issue create + first transition), history ordering is **not guaranteed** (sort yourself), field names are inconsistently cased, and **status→status-category mapping is absent** (fetch `/rest/api/3/status` separately).

**Story points** are a custom field with no fixed ID. Discover via `GET /rest/api/3/field` (match name "Story Points" / "Story point estimate" for team-managed) or, more reliably, the **board's estimation field** from `GET /rest/agile/1.0/board/{id}/configuration`. Sprint scope/velocity comes from `/rest/agile/1.0/board/{id}/sprint` + `/rest/agile/1.0/sprint/{id}/issue`; there is **no supported velocity-report endpoint** (only the private `/rest/greenhopper/1.0/rapid/charts/velocity`), so commit-vs-complete must be computed from sprint membership changelog.

**Rate limits**: points-based, ~65,000 points/hr shared (Tier 1) up to 500k (Tier 2); search of 50 issues ≈ 51 points. Honor `Retry-After` on 429, watch `X-RateLimit-*`/`RateLimit-Reason`, back off exponentially with jitter. **Auth**: build ONE distributable **OAuth 2.0 (3LO)** app (or **Connect/Forge** for marketplace) — collecting per-user API tokens violates Atlassian cloud-app security policy. **Sync**: webhooks are at-least-once and dynamic ones expire in 30 days (refresh via PUT), so the correct design is **webhook + scheduled `updated >= lastSync` JQL reconciliation backfill**.

**Recommendations**

## Recommended incremental ingestion design (adopt)

**1. Auth — one distributable OAuth 2.0 (3LO) app.** For a Claude Code plugin shipped to teams, build a single 3LO app with scopes `read:jira-work`, `read:jira-user`, `read:board-scope:jira-software`, `read:sprint:jira-software`, `offline_access` (for refresh tokens). Do **not** collect per-user API tokens or spawn per-customer apps — it violates Atlassian cloud-app security policy and won't scale. (Consider a **Forge** app only if you want Marketplace distribution and Atlassian-hosted egress.)

**2. Field discovery on connect.** On first sync per site: call `GET /rest/api/3/field` and cache the map; resolve the story-point field from each **board's** `GET /rest/agile/1.0/board/{id}/configuration` estimation field (more reliable than name matching, handles team-managed `Story point estimate`). Cache `GET /rest/api/3/status` for the status→category map and refresh it daily.

**3. Backfill (cold start).** Page through `POST /rest/api/3/search/jql` with `ORDER BY created ASC`, `fields` limited to what you need, looping on `nextPageToken` until it is absent. For each issue page, pull full history via **`POST /rest/api/3/changelog/bulkfetch`** (≤1000 issues, filter to `status` + Sprint + story-point fields) — never rely on inline `expand=changelog`, which truncates at 100 histories and will silently corrupt cycle-time on old issues. Use `/search/jql/approximate-count` only for progress UX, not correctness.

**4. Incremental delta (warm path).** Run a scheduled job every N minutes: `JQL = updated >= "-{N+buffer}m" ORDER BY updated ASC`, page via `nextPageToken`, and re-pull changelog deltas for changed issues. Persist a per-site `lastSyncCursor` = max(`updated`) seen, with an overlap buffer (e.g., 2× interval) to absorb **eventual-consistency lag** in the search index. Use the **`reconcileIssues`** param for issues you know changed (from webhooks) but that the index hasn't surfaced yet.

**5. Webhooks as a latency accelerator, not source of truth.** Register dynamic webhooks (`POST /rest/api/3/webhook`) for `jira:issue_created/updated/deleted` with a `jqlFilter` scoping to tracked projects. Remember the **30-day expiry** → a cron must `PUT` to refresh, and OAuth apps are capped at **5 webhooks/user**. Because delivery is **at-least-once and lossy**, treat webhooks as triggers that enqueue an issue for reconcile; the scheduled `updated`-JQL job remains the authoritative safety net.

**6. Rate-limit discipline.** Budget against ~65k points/hr (search of 50 issues ≈ 51 points). Request minimal `fields`, cap concurrency, honor `Retry-After` and watch `X-RateLimit-NearLimit`; implement exponential backoff (2s base, ×2, cap 30s, ±30% jitter, 4 retries). Cache with ETags where supported.

## Avoid
- Do **not** use deprecated `/rest/api/3/search` or assume `total`/`startAt` exist — they're gone in Cloud.
- Do **not** trust inline `expand=changelog` for backfill (100-history cap).
- Do **not** compute flow metrics from status without the status→category map, and do **not** assume changelog ordering.

## Do better than competitors
- **Honest flow efficiency:** ship a per-team active-vs-queue status classifier (AI-assisted, human-confirmed) so flow-efficiency is meaningful rather than hidden — most tools skip this.
- **Membership-accurate velocity:** reconstruct sprint scope from the Sprint-field changelog to correctly show committed-vs-completed and mid-sprint scope creep, instead of the final-membership snapshot most integrations use.
- **Self-healing sync:** surface an Index Freshness metric and auto-reconcile, so customers never see silently-dropped transitions — a common, invisible failure in naive `updated`-poll integrations.

## DC/Server portability note
If you later support Jira Data Center, abstract the search layer: DC still uses `/rest/api/2/search` with `startAt`/`total` (no `/search/jql`, no nextPageToken), basic-auth/PAT instead of 3LO/Forge, no points-based rate limiting, and webhooks are admin-configured (no per-app dynamic registration). The changelog/Agile models are otherwise compatible.

## ⚠️ Notes
- **Secrets:** OAuth client secret and refresh tokens must be stored encrypted server-side, never in the plugin client or logs.
- **Scope minimization:** request read-only scopes only; an analytics product should never hold write scopes.
- **PII/ethics:** changelogs expose per-individual activity; aggregate to team level for delivery metrics and avoid surfacing individual surveillance-style rankings (gaming + ethical risk on every per-person metric above).

### 2.13 AI / Probabilistic Metric Design (LLM-computed soft delivery metrics from GitHub + Jira)

This dimension covers the five "soft" delivery metrics that cannot be computed by counting or diffing alone and require an LLM judge: Ticket-Work Alignment, Effort Proportionality, Work Classification, Velocity Anomaly Explanation, and PR Quality Score. The defensible architecture is **hybrid in every case**: deterministic features (diff stats, file paths, conventional-commit prefixes, cycle-time z-scores, presence of tests) are computed first and fed to the LLM as evidence, and the LLM is constrained to a rubric-driven verdict with citations back to that evidence. The LLM is never the sole source of a number it could fabricate.

**LLM-as-judge methodology is well-studied and the findings are blunt.** Position bias is real and volatile: across 15 judges / ~150k evaluations, position consistency ranged 0.23–0.82 and preference fairness swung -0.13 to +0.32, with the direction flipping between tasks for the *same* judge (arXiv 2406.07791, AACL 2025). The dominant mitigations are: prefer **pointwise rubric scoring** (which structurally avoids position bias) over pairwise; use **low-precision scales** (binary or 1–5, never 1–10 — judges cluster on 6s and 7s); **temperature 0**; **constrained decoding** so output is schema-valid by construction (Anthropic structured outputs compile the JSON schema into a grammar that restricts token sampling — guaranteed valid JSON, type-safe, no parse retries); and **calibrate against a human-labelled gold set**, iterating the rubric until agreement (e.g. Cohen's κ) closes. Majority voting across model families is effective for >95% of instances where bias persists.

**Per-metric grounding.** Work classification maps cleanly onto Conventional Commits (feat/fix/refactor/test/docs/chore/perf/style) and Swanson's corrective/perfective/adaptive taxonomy; keyword-only baselines hit only ~46% F1, so the diff must be in the prompt, not just the message. Effort estimation: zero-shot LLMs already beat supervised Deep-SE on Jira title+description (Spearman/Pearson ~0.40), few-shot scale-aware selection lifts that ~0.08–0.12, and crucially **LLMs preserve relative ordering far better than absolute magnitude** — so frame effort as a proportionality ratio/ordinal band, never a raw point prediction. Ticket-Work Alignment and PR Quality are rubric-scored faithfulness/quality judgments where the chief risk is hallucinated coverage claims; mitigate with mandatory evidence quotes and an "insufficient information" escape hatch. Velocity anomaly explanation is a generative-attribution task constrained to a closed menu of candidate causes ranked against deterministic signals (WIP, reviewer latency, ticket churn) to prevent narrative confabulation.

**Auditability is the product moat.** Every verdict must persist: prompt template version, model ID + snapshot, temperature, the exact deterministic feature vector shown, the structured verdict with per-claim evidence pointers, and a self-reported confidence that is itself validated against gold-set accuracy. This makes scores reproducible, contestable by engineers, and safe to surface in a Flow-style dashboard without inviting weaponisation.

**Recommendations**

## What THIS product should adopt

**1. Make every soft metric hybrid, never pure-LLM.** Compute a deterministic feature vector first (diff stats, conventional-commit prefix, cycle-time z-score, test-file presence, review latency) and pass it to the LLM as *evidence to reason over*, not as something to estimate. The LLM's job is rubric judgment and attribution, never arithmetic it could hallucinate. This is the single biggest correctness and trust lever.

**2. Pointwise rubric scoring, low-precision scales, temperature 0, constrained decoding.** Position bias (consistency as low as 0.23, direction flips per-task) makes pairwise judging unreliable for this domain — judge each artifact against an absolute rubric. Use 0–4 / 0–2 ordinal bands or binary, never 1–10. Use Anthropic structured outputs (`output_config.format`, grammar-constrained) so output is schema-valid by construction — no JSON-repair loop, type-safe, auditable.

**3. Mandatory evidence quotes + an 'insufficient information' escape hatch on every claim.** This is the core anti-hallucination mechanism. A criterion can only be 'covered', a cause only 'likely', a PR only 'well-described' if the model quotes the exact diff hunk / signal / sentence justifying it. No quote => 'unclear' / 'insufficient signal'. For velocity explanation, constrain causes to a closed menu — never free-form narrative.

**4. Ship a calibration harness as a first-class feature.** Maintain a human-labelled gold set per customer/team; report agreement (Cohen's κ, macro-F1 for classification, Spearman for effort ordering) and iterate rubrics until κ≥0.6. Validate the model's self-reported confidence against actual gold-set accuracy and recalibrate — an uncalibrated confidence is worse than none. Persist for every verdict: prompt-template version, model ID+snapshot, temperature, the feature vector shown, the structured verdict with per-claim evidence, and confidence. This audit record is the product's defensibility and lets engineers contest a score.

**5. Effort = ordinal proportionality, never a points number.** LLMs rank far better than they estimate magnitude. Output {much_lower…much_higher vs expected} with a log-ratio, cross-checked against a deterministic cycle-time z-score. Exempt research/spike issue types entirely.

**6. Use majority voting across two model families only where it pays.** For high-stakes or low-confidence verdicts, ensemble across families resolves >95% of residual bias — but it doubles cost, so gate it on confidence rather than running it everywhere.

## What to avoid (vs competitors like Flow)

- **No individual-level leaderboards from these metrics.** They measure ticket hygiene and process as much as a person; SPACE authors are explicit that productivity metrics must not be weaponised per-developer. Surface team/repo aggregates and trends.
- **Don't let effort or PR-size become a target** — lines-changed and commit-count are trivially gamed (Goodhart); present them as descriptive context, flag large PRs rather than hard-penalise.
- **Don't trust Jira issue type or PR body as ground truth** — both are routinely mislabelled/padded; that's exactly why the diff must be in the prompt.
- **Differentiator:** Flow shows raw charts; this product should ship *cited, contestable explanations* (anomaly attribution with evidence pointers, per-criterion alignment) — the auditability is the moat.

### 2.14 Ethics, anti-patterns & Goodhart risks

Developer-productivity measurement is an ethics minefield governed by **Goodhart's Law**: "when a measure becomes a target, it ceases to be a good measure." Every proxy (LOC, commits, story points, even DORA metrics) gets gamed once tied to evaluation — not through dishonesty but through rational optimization by the measured. The DORA team itself names this explicitly, warning against "setting metrics as a goal" and "one metric to rule them all," and stresses metrics must drive *continuous improvement*, never serve as performance targets or inter-team competition.

The **2023 McKinsey controversy** is the central cautionary tale. McKinsey's Aug-2023 article claimed developer productivity *can* be measured at the individual level via metrics like **Inner/Outer Loop time** (push 70% of time into "inner loop" coding), **Contribution Analysis** (per-developer contribution to backlog), **Developer Velocity Index (DVI)**, and **Talent Capability Score**. Four of five metrics measured effort/output not outcome/impact, and "Contribution Analysis" was a red flag for individual surveillance and layoff selection. The expert backlash was severe: Gergely Orosz & Kent Beck (Pragmatic Engineer) warned "the act of measurement changes how developers work as they try to game the system," citing Facebook's survey scores degrading once tied to comp/layoffs. Critics noted McKinsey leaned on **Nicole Forsgren's** DORA/SPACE research without citing her. Dan North and Jacob Kaplan-Moss reinforced that individual measurement is intractable and harmful.

**Framework authors are explicit on ethics.** SPACE (Forsgren et al., ACM Queue 2021): activity metrics "should never be used in isolation either to reward or to penalize developers"; SPACE is for "team and organizational insights, not individual performance evaluation"; always balance perceptual (survey) + system metrics across multiple dimensions. **DevEx** (Noda, Storey, Forsgren, Greiler 2023) reframes around feedback loops, cognitive load, flow state — friction not output.

**Stack/forced ranking** is the most documented backfire: Microsoft abandoned it in 2013 after it destroyed collaboration (top engineers avoided strong teams to dodge unfavorable curves); Amazon retreated from scrutiny-driven systems after burnout/turnover. Leaderboards and per-developer rankings reliably erode psychological safety and cross-team help.

**GDPR/privacy**: per-developer performance data is personal data; systematic performance profiling is an Art. 35 DPIA trigger. Employee **consent is not a valid legal basis** (power imbalance); employers must rely on documented **legitimate interest** balancing tests *done before* monitoring — regulators treat undocumented claims as no legal basis. Principles: data minimization, purpose limitation, transparency. **Design guardrails**: make per-developer data self-view only and growth-oriented; reserve all comparison/ranking for team-or-higher aggregates; never feed metrics into reviews/comp/layoffs; require multiple balanced metrics; show methodology transparently; and prefer outcome/experience signals over vanity output counts.

## Design guardrails this product should adopt

**1. Hard architectural separation: individual data is self-view only.**
- Per-developer metrics are visible to that developer alone, framed for personal growth, compared only against their *own* history — never percentiles, never leaderboards, never a manager-facing per-developer dashboard. Enforce via access control, not policy alone. This aligns with GDPR subject-access rights while structurally preventing surveillance.
- Reserve all comparison, ranking, and benchmarking for **team-or-higher aggregates**. Make it technically impossible to drill from a team aggregate down to an individual ranking.

**2. Refuse to build the anti-patterns — and say so.**
- No stack ranking, no individual leaderboards, no LOC/commit-count scorecards. Treat "we don't do this" as a marketing differentiator backed by Microsoft/Amazon evidence and Forsgren/DORA guidance.
- LOC and raw activity counts appear (if at all) only as team-level health/trend signals labeled as such, never as productivity scores.

**3. Bake in anti-Goodhart by design.**
- Default to **multiple balanced metrics** (SPACE-style: pair system metrics with perceptual survey data; cover >=2 dimensions). Surface a balanced scorecard, never a single composite "productivity number."
- Lead with **DevEx friction signals** (review latency, CI duration, context-switching, cognitive load) and **DORA team delivery metrics** as diagnostics for improvement — explicitly framed as "find bottlenecks," not "hit targets." Show DORA's own Goodhart warning in-product when a user tries to set a metric as a goal.

**4. Transparency and consent as product features.**
- Show methodology/formula for every metric in-line ("how is this computed?"). No black-box scores.
- Ship a built-in **DPIA template and legitimate-interest assessment** plus a transparency notice generator. Default retention limits, purpose limitation, and per-tenant data isolation (never cross-client benchmarking — a confidentiality and GDPR risk). Treat employee consent as invalid; guide buyers to documented legitimate interest.
- Provide an admin "ethics mode" that is on by default: blocks export of individual data into review/comp workflows and warns when usage drifts toward evaluation.

**5. Outcome over output.**
- Bias the headline metrics toward outcome/impact and developer experience (the dimension McKinsey ignored), not effort proxies. This is the core positioning against output-counting competitors: measure the *system and the friction*, empower individuals privately, and never weaponize the data against the people generating it.

**Caveat (org policy):** This is product/ethics scaffolding, not legal advice — buyers must route DPIA and legitimate-interest assessments through qualified data-protection counsel.

### 2.15 Data modeling & analytics architecture

## Recommended approach

For a self-hostable, open-source Pluralsight Flow competitor sourced from GitHub + Jira and shipped as a Claude Code plugin, the durable winning pattern is **a normalized OLTP core in Postgres (with SQLite as the zero-config default for single-team installs), feeding precomputed daily rollups, and an optional embedded DuckDB engine attached to the same files for heavy ad-hoc analytics.** This mirrors the architecture every credible tool has converged on: Google's open-source **Four Keys** ingests raw webhook events into one `events_raw` table then derives `changes`/`deployments`/`incidents` via scheduled SQL ([Google Cloud](https://cloud.google.com/blog/products/devops-sre/using-the-four-keys-to-measure-your-devops-performance)), and DORA/SPACE/DevEx all assume an event-sourced substrate.

## Schema philosophy

Model two layers. **(1) A normalized "system-of-record" layer** that faithfully mirrors GitHub and Jira objects (`organisations`, `repositories`, `contributors`, `commits`, `pull_requests`, `reviews`, `review_comments`, `deployments`, `jira_projects`, `issues`, `issue_transitions`, `sprints`, `sprint_issues`). Keep upstream IDs as natural keys (`github_node_id`, Jira `issue.id`) with a surrogate BIGINT PK; store the full raw payload in a `JSONB`/`TEXT` column per row so re-derivation never requires re-fetching. **(2) A derived metrics layer**: append-only `sync_state` cursors, an immutable `events` stream, and `metric_snapshots` time-series rollups. Crucially, `issue_transitions` must be its own append-only table reconstructed from the Jira changelog (`expand=changelog` → `histories[].items[]`), because Jira stores no time-in-status field and history ordering is not guaranteed ([Atlassian](https://support.atlassian.com/jira/kb/how-to-analyze-the-history-or-changelog-of-an-issue-in-jira/), [Mike Bowler](https://blog.mikebowler.ca/2024/04/09/jira-issue-history/)). Likewise PR stage timestamps (first-commit, opened, first-review, approved, merged) should be denormalized onto `pull_requests` as computed columns so cycle-time queries don't re-walk events.

## Identity stitching

Use a three-tier model: `identities` (one row per raw account fingerprint — a GitHub login, a commit author email, a Jira accountId) all pointing via FK to a canonical `persons` table. Stitch deterministically first (verified email match across GitHub `users.email`/commit email and Jira `emailAddress`), then heuristically (normalized name + email-local-part), and surface low-confidence merges for human confirmation. This matches how Git Integration for Jira resolves commit author email → Jira user, falling back to raw name when no match ([GitKraken](https://help.gitkraken.com/git-integration-for-jira-data-center/git-user-identity-gij-self-managed/)).

## Storage & sync

Track sync with per-resource high-water marks (`updated_at` cursor + GraphQL endCursor), advance the watermark only after the load commits, and re-read a small overlap window to catch late writes, relying on idempotent UPSERTs ([dlt](https://dlthub.com/docs/general-usage/incremental-loading), [Fivetran](https://www.fivetran.com/blog/building-efficient-data-pipelines-with-incremental-updates)). Use plain SQL migrations (golang-migrate / Alembic-style up/down files) checked into the repo.

## What this product should adopt

**1. Two-layer schema, raw payload retained.** Mirror GitHub/Jira objects in a normalized OLTP core and keep the full raw JSON per row (`raw JSONB`). This is exactly the Four Keys `events_raw`-then-derive pattern and means metric definitions can change without re-fetching from rate-limited APIs — a decisive advantage over competitors that discard raw data. Re-derivation is just re-running SQL.

**2. Core schema (Postgres DDL sketch).**
- `organisations(id PK, github_login, jira_cloud_id, ...)`
- `repositories(id PK, org_id FK, github_node_id UNIQUE, default_branch, ...)`
- `persons(id PK, display_name, primary_email_hash)` ← canonical human
- `identities(id PK, person_id FK NULL, kind ENUM('github_login','commit_email','jira_account'), external_id, confidence, raw)` ← stitch table
- `contributors` collapses into `identities`; reference `person_id` everywhere a metric needs "who".
- `commits(sha PK, repo_id FK, author_identity_id FK, committer_identity_id FK, authored_at, committed_at, additions, deletions, raw)`
- `pull_requests(id PK, repo_id FK, number, author_identity_id FK, created_at, merged_at, merged_by_identity_id, first_commit_at, first_review_at, approved_at, state, raw)` — stage timestamps denormalized.
- `reviews(id PK, pr_id FK, reviewer_identity_id FK, state, submitted_at, raw)`; `review_comments(id PK, pr_id FK, review_id FK NULL, author_identity_id FK, created_at, raw)`
- `deployments(id PK, repo_id FK, sha, environment, status, created_at, finished_at, raw)`
- `jira_projects`, `issues(id PK, project_id FK, key, type, status_id, status_category, story_points, assignee_identity_id FK, created_at, resolved_at, raw)`
- `issue_transitions(id PK, issue_id FK, from_status_id, to_status_id, transitioned_at, actor_identity_id FK)` ← append-only, rebuilt from changelog, **sorted by timestamp on ingest**.
- `sprints(id PK, board_id, name, state, start_at, end_at, complete_at)`; `sprint_issues(sprint_id FK, issue_id FK, points_at_start, added_after_start BOOL, PK(sprint_id,issue_id))` — **snapshot membership at sprint start** since Jira boards keep no history.

**3. Identity stitching: deterministic-first, reversible, auditable.** Email-verified matches auto-merge (confidence 1.0); fuzzy matches stay split and surface in a review queue. Store `confidence` and never hard-delete an identity (set `person_id` nullable so unmerges are trivial). Handle GitHub `noreply` emails as un-matchable. Treat all of this as PII (hash emails, support deletion).

**4. Snapshots vs on-the-fly: do both, deliberately.** Compute DORA/cycle-time **on-the-fly from the normalized core** for any date range (cheap with proper indexes / DuckDB). Additionally write **immutable daily `metric_snapshots(scope_type, scope_id, metric, day, value, window)`** for fast dashboards and to freeze historical values against late-arriving data. Never store per-person productivity rankings as first-class snapshots — that invites misuse and contradicts SPACE's explicit warning against single-number individual metrics.

**5. Sync state: per-resource watermarks + overlap window.** `sync_state(source, resource, scope_id, cursor, last_run_at, status, error)`. Advance the cursor only after the batch commits; on each run re-read a configurable overlap window (e.g. 2x poll interval) before the watermark and rely on idempotent `INSERT ... ON CONFLICT DO UPDATE` keyed on upstream node IDs. Jira changelogs mutate old issues, so its overlap must be generous. Expose watermark lag as a health metric.

**6. Database choice — ship SQLite by default, support Postgres, attach DuckDB for analytics.**
- **SQLite** = zero-config default for single-team self-hosters; perfect for the transactional UPSERT-heavy sync path.
- **Postgres** = opt-in for multi-team/multi-user serving (DuckDB and SQLite are single-writer/single-process and won't handle concurrent dashboard load).
- **DuckDB** = embedded analytics engine; attach it to the SQLite file (or read Parquet exports) for the 20-50x-faster aggregations that power large date-range dashboards, without adding a server dependency. Avoid making DuckDB the system of record (single-process, weak concurrency).
Keep the schema SQL-portable (avoid Postgres-only types in the core; use TEXT for JSON on SQLite, JSONB on Postgres via a dialect shim).

**7. Migrations.** Plain checked-in SQL up/down files run by an embedded migrator (golang-migrate if Go, Alembic/Drizzle if Python/TS), version table in-DB, forward-only in production. Ship the same migration set for SQLite and Postgres with a thin dialect layer.

## What to do better than competitors
- **Retain raw payloads** so metric definitions are revisable retroactively (most tools can't re-derive).
- **First-class, reversible identity model** with confidence and a human review queue — competitors often silently mis-merge.
- **Snapshot freezing + on-the-fly recompute** so historical numbers are stable yet definitions remain auditable.
- **Treat individual metrics as a liability**: default dashboards to team/repo scope, gate person-level views, and document gaming/ethics inline — a differentiator versus surveillance-flavored incumbents.

## ⚠️ Security & ethics notes
- Emails/accountIds are PII (GDPR) — hash at rest, support per-person deletion, and document retention.
- Identity mis-stitching can produce defamatory individual stats; keep every merge reversible and audited.
- DORA/cycle-time numbers are trivially gamed (PR splitting, unfiled incidents, rubber-stamp approvals); pair every metric with its caveat in the UI and never use them for individual performance review (per SPACE).
- API tokens for GitHub/Jira are secrets: store via OS keychain/env, never in the DB or repo.

Ship ONE plugin (lazy-flow) via a git-hosted marketplace, bundling skills (slash commands like /lazy-flow:dora), agents for AI insights, and one TypeScript MCP server in .mcp.json that owns all GitHub+Jira access and the shared metrics DB. Make install trivial and team-wide by committing extraKnownMarketplaces and enabledPlugins to the repo .claude/settings.json (project scope) so members are auto-prompted on folder-trust; for org-wide enforcement use managed settings with strictKnownMarketplaces allowlisting only your repo (users cannot override), and CLAUDE_CODE_PLUGIN_SEED_DIR for containers. For transparency and auditability, compute deterministic metrics (DORA throughput, PR cycle time, review latency, WIP) inside the MCP server against a pre-aggregated shared DB and return them via registerTool with outputSchema and structuredContent so values are exact and machine-checkable; expose dashboards/snapshots as Resources for at-mention; have a bundled subagent produce probabilistic narratives from those deterministic outputs so the model only narrates. For shared data and credentials, run the MCP server as a remote Streamable HTTP backend (single ingestion path, one shared DB, auto-reconnect) rather than N local stdio servers that drift; authenticate users via per-user OAuth or a userConfig sensitive token stored in the keychain; never ship the raw DB DSN or a broad GitHub/Jira PAT in client config (the server holds those and exposes only scoped tools); CLAUDE_PLUGIN_DATA persists state across updates. Versioning: set version in exactly one place (prefer the marketplace entry, or omit for the commit SHA), bump on every metric-definition change, and offer stable/latest channels via two ref-pinned marketplaces. Avoid client-held org PATs/DSNs, double-declaring version, relative-path sources behind URL-only marketplaces, and stdio for the shared data path.
