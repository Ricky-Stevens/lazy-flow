# lazy-flow — Adversarial Review v1 (findings & resolution)

> Produced by a 9-dimension adversarial workflow (19 agents, ~1.9M tokens) on 2026-06-06: each dimension assumed broken and proved failures; a refutation pass killed weak findings; synthesis prioritized the rest. `SPEC.md`/`WORKPLAN.md` v2 incorporate these. This is the audit trail.


**Verdict:** fit-for-purpose = `with-fixes`


The deterministic-catalogue concept and the metric research are strong, but as written the SPEC ships three load-bearing promises the architecture cannot keep — "identical data/metrics on every install," "ranking is architecturally impossible," and "no install step / single bundled server.js" — each contradicted by the design itself (per-user API access scoping, public per-person snapshots readable by all peers, and better-sqlite3 being a native addon). On top of these, two correctness primitives the product is built around (the per-workflow Flow State Model and Jira board/started-done boundaries) are depended upon but have no producing work item or ingestion source, and a cluster of metric formulas will emit NaN/Infinity, negative percentages, or wrong numbers on ordinary day-one inputs. None of these require architectural redesign that the spec hasn't already sketched (the §5.4 shared-ingester seam, additive guards, classification layers), but the spec must be corrected and several first-class work items added BEFORE fan-out, because the WORKPLAN's golden-test "exact match" gate is currently testing under-specified targets. Fit-for-purpose with fixes — the bones are good, the promises are oversold and the engine contract (§8.6) is under-pinned.


## Critical themes

- §8.6 is the engine-version/pinning contract but pins only three things (complexity-operator counting, churn window, DORA bands) while the product's determinism/reproducibility/auditability thesis requires it to ALSO pin percentile method, zero/empty-input semantics, reopen/Done dedupe, squash-aware lead-time anchoring, temporal status-classification versioning, HALOC file classification, PRNG/sample-order, and reporting timezone. Under-pinning here is the single highest-leverage root cause: it makes the golden 'exact match' gate test under-specified targets.
- The local-first per-user inversion of the research's single-ingester recommendation (D1) silently breaks three headline claims at once: §5.3 'identical data' (false under per-user GitHub/Jira access scoping AND per-install timing/engine-version skew), the N× multiplication of API rate-limit pressure / LLM cost / credential sprawl, and the unbuildability of cross-team org rollups. The §5.4 shared read-only snapshot artifact is the available fix and should be promoted from optional to the recommended team path.
- Two correctness primitives the product markets as its moat are depended-upon but have NO producing work item and NO ingestion source: the per-workflow Flow State Model (flow_state_models.workflow_id is an orphan FK; no workflow discovery; AI-seeded in Wave 5 but consumed in Wave 4 — a backward DAG edge) and the Jira board started/done column boundary (never ingested, so cycle-time/WIP/aging have no defined start). These are the headline 'correct flow metrics' capability and half of it is unbuilt.
- Pervasive degenerate-input and statistical-soundness gaps will mislead or error on ORDINARY day-one inputs: division-by-zero across ~8 ratio metrics (NaN/Infinity on a holiday week / new sprint / binary-only PR), unpinned percentile estimator + no minimum-sample floors, negative 'predictability' from unbounded 1−CV, reopen/Done double-counting, and subtask-point double-counting of velocity. All fixable with additive guards + golden tests, but all currently shippable defects.
- The flagship ethical/privacy posture is self-contradicting and ships the riskiest configuration by default: 'ranking architecturally impossible' is false while per-person snapshots are persisted and readable by every installed peer under the public default; the public-by-default to unknown OSS adopters almost certainly fails GDPR legitimate-interest balancing with only documentation (not enforcement) as mitigation; and 'hashed at rest' presents a trivially-reversible unsalted email hash as a privacy control.
- The packaging contradiction is fatal as written: better-sqlite3 (a node-gyp native, ABI-locked addon) cannot be inlined into a single bundled server.js, so 'no install / no native build / single server.js' (§12.3, §17 AC#1) is factually impossible with the chosen stack; the first user on a different OS/arch/Node-ABI gets a crash on boot. The WASM tree-sitter grammars have the same unspecified asset-resolution problem.


## Prioritized refinements (upheld/downgraded)


### P1 · [CRITICAL] Packaging contradiction: better-sqlite3 native addon cannot be a single bundled server.js with no install step

**Problem:** §12.3/D12/§17 AC#1 promise 'single bundled server.js, no runtime npm install, no native build on host' while D5/WP-STORE pin the store on better-sqlite3, a node-gyp-compiled ABI-locked native addon that bundlers cannot inline into a .js text artifact. The first teammate on a different OS/arch/Node-ABI gets 'Cannot find module better_sqlite3.node' or a NODE_MODULE_VERSION mismatch crash on boot. The author already noticed native builds are bad (the §12.3 'tree-sitter uses WASM → no native build' note) but missed that better-sqlite3 IS one. The §12.3 WASM grammars have a parallel unspecified problem: web-tree-sitter loads tree-sitter.wasm + grammar .wasm as separate runtime assets via locateFile, never bundled into server.js and with no copy-step or install-dir-relative resolution specified, so grammar loads fail and all complexity metrics error.

**Fix:** Decide and write down the SQLite distribution mechanism: (a) switch default store to a bundleable WASM SQLite (wa-sqlite / @sqlite.org/sqlite-wasm) so the artifact is truly ABI-independent — preferred, also fixes the ABI-brick and Node-version-pin problems; OR (b) use node:sqlite (Node 22+ built-in) with a pinned min Node; OR (c) drop the single-file claim and ship a full node_modules + multi-platform prebuild matrix (prebuild-install/node-pre-gyp). Separately: add an explicit tsup/esbuild copy-assets step placing tree-sitter.wasm + every grammar .wasm under server/dist/grammars/, set web-tree-sitter locateFile to resolve against fileURLToPath(import.meta.url) (not cwd), pin the web-tree-sitter version, and add a clean-machine (no node_modules) cross-ABI boot+handshake smoke test to CI as a Wave-0 gate.

**Target:** SPEC §12.3, D5, D12, §17 AC#1; WORKPLAN WP-STORE, WP-TOOLING, WP-CODE-ANALYSIS, WP-MCP-SERVER; new Wave-0 CI boot-smoke gate


### P2 · [CRITICAL] Per-workflow Flow State Model is depended-upon but has no producing work item and no ingestion source (orphan workflow_id; Wave-5→Wave-4 backward edge)

**Problem:** flow_state_models (§6.2) is keyed on workflow_id and C3 names the per-workflow status→active/wait map as where flow-efficiency accuracy lives or dies, yet: (1) no WP discovers Jira workflows/workflow-schemes or resolves issue→(project,issuetype)→workflow, so workflow_id is an orphan FK with no populating source (RESEARCH line 462 lists workflow+status definitions as a required source); (2) no WP builds the model itself (deterministic seed + LLM seed + human-confirm queue + confidence scoring + config persistence) — WP-DB-SCHEMA only declares the table; (3) WP-METRICS-FLOW (Wave 4) consumes it but AI seeding is Wave 5 — a backward DAG edge. A Wave-4 agent reads an empty table and either errors or silently applies a wrong global map — the exact silent-false-dashboard the product exists to prevent (C3: 'a global active/wait map is wrong for most teams').

**Fix:** Add WP-FLOWSTATE-MODEL (deterministic seeding heuristic + LLM seed + human-confirm queue + confidence scoring + config persistence + a review skill + a deterministic implicit-wait fallback flagged low-confidence) and schedule its deterministic+seed parts BEFORE WP-METRICS-FLOW. Add workflows/workflow_scheme_mappings tables (or a derived project+issuetype→status-set key) and a WP-JIRA-WORKFLOW Wave-3 ingestion item. Define resolution issue→workflow→flow_state_models row; make WP-METRICS-FLOW depend on both; parameterise the Flow Efficiency golden expectation on the confirmed model.

**Target:** SPEC §6.2, §1 C3, §8.2; WORKPLAN — new WP-FLOWSTATE-MODEL + WP-JIRA-WORKFLOW, WP-DB-SCHEMA, WP-METRICS-FLOW, §1 DAG


### P3 · [CRITICAL] Jira board configuration (started/done columns) never ingested — cycle time, WIP, aging WIP, throughput have no defined start boundary

**Problem:** RESEARCH lines 909/925/927/929 name /rest/agile/1.0/board/{id}/configuration as the REQUIRED source for the started/done column boundary of Cycle Time, WIP, Throughput and Aging WIP. §7.2 fetches boards/sprints and reads the story-point field from board config but never ingests the column→status mapping; §6.1 has no board_columns table; §8.2 leaves cycle-time start undefined. status_category does NOT suffice (RESEARCH explicitly rejects it): Jira's coarse In-Progress category groups queue columns (Selected-for-Dev) with active columns (In-Dev), so category-only start lines inflate cycle time and over-count WIP for most teams. Raw retention does not help because the board-config resource is never fetched.

**Fix:** Add a board_configs/board_columns table (board_id, column_name, status_ids[], is_started_col, is_done_col) ingested from /rest/agile/1.0/board/{id}/configuration; define cycle-time start as first entry into a status mapped to a started column per board; add WP-JIRA-BOARDCONFIG to Wave 3 and make WP-METRICS-FLOW depend on it with golden fixtures for queue-vs-started columns.

**Target:** SPEC §6.1, §7.2, §8.2; WORKPLAN — new WP-JIRA-BOARDCONFIG, WP-METRICS-FLOW


### P4 · [CRITICAL] metric_snapshots 'immutable' contradicts the late-arrival/reconciliation sync model — two product surfaces report different numbers for the same day

**Problem:** §6.2 calls metric_snapshots 'immutable daily rollups'; WP-SNAPSHOTS asserts 'snapshot == on-the-fly recompute'; but §7.3 reconciliation is the completeness guarantee that backfills missed/late data AFTER the day boundary and §7.4 is last-writer-wins. These cannot all hold: a frozen day-D snapshot permanently disagrees with recompute once late events for day D land, and §15 has dashboards read snapshots while /explain recomputes — so get_dora and /explain contradict each other for the same day and a DORA band can flip. No watermark/invalidation/versioning scheme exists. Compounded by the absence of an engine_version column: a graph mixing v1-frozen days with v2-recomputed days (after a formula change bumps the engine version per §8.6/§10) is silently undetectable — the exact false-trend the product claims to prevent.

**Fix:** Make snapshots versioned not immutable: key on (scope, metric, day, ingest_watermark_version) and store the watermark + engine_version each snapshot was computed against. On reconciliation that mutates any raw row for day D, mark day-D snapshots stale and lazily recompute. Tools return latest version + as_of/recomputed_at and refuse to plot across mixed engine versions without an explicit flag. Change WP-SNAPSHOTS acceptance to 'recompute of a CLOSED window (watermark stable) equals the snapshot' and define a window-closing/grace-period rule. Add WP-REDERIVE for engine-version-bump invalidation + a version-boundary continuity test.

**Target:** SPEC §6.2, §7.3, §7.4, §8.6; WORKPLAN WP-SNAPSHOTS, WP-RELEASE, new WP-REDERIVE


### P5 · [CRITICAL] 'Identical data and metrics on every install' (§5.3) is false under per-user API access scoping AND per-install timing/version skew

**Problem:** §5.3 states unconditionally that two developers syncing the same org derive identical data and metrics, and this thesis is the foundation of the transparency/auditability moat. It is structurally false: (1) Jira 3LO is per-user consent and GitHub PAT-fallback / App repo-selection differ per principal, so each install computes team/org aggregates over different denominators while both display identical formula_doc/trust_tier/as_of badges — verified across the Metrics, Data Model, Ingestion, and Architecture dimensions; (2) even with identical access, §6.2 immutable per-install snapshots + §7.5 per-machine freshness + §12.4 engine-version skew via /reload-plugins mean two installs legitimately disagree on the same scope/window with no canonical authority — defeating contestability (contesting a number means contesting one of N copies). There is no coverage/access-scope dimension anywhere in the schema or tools.

**Fix:** Qualify §5.3 to 'identical only for resources every member's credential can read, at equal sync watermark and engine version.' Add a coverage_fingerprint (hash of accessible repos/projects/issue-types) + watermark + engine_version to metric_snapshots and every tool's data_quality; emit a completeness flag on every aggregate ('computed over 19/20 repos visible to your credential'); refuse cross-install comparison unless fingerprints/versions match. Preferred structural fix (resolves this + priority 6): make the §5.4 single App-authenticated ingester publishing a signed read-only snapshot the recommended team path, with local-first as the solo path.

**Target:** SPEC §5.3, §5.4, §6.2, §7.5, §11; WORKPLAN WP-VISIBILITY, WP-SNAPSHOTS


### P6 · [CRITICAL] Local-first multiplies API rate-limit pressure, LLM cost, and credential exposure by developer count (N×), and the budgets/cost-model are written as if there is one ingester

**Problem:** RESEARCH designed ingestion for a SINGLE server-side ingester (lines 200-214 '(server-side)', 1092 token-per-installation pool, 1466 'remote Streamable HTTP backend rather than N drifting local stdio servers'); D1 inverted this to per-user local-first and §7 carried the per-installation numbers verbatim without re-modeling N concurrent installers. Consequences verified across three dimensions: (a) GitHub limits are per-installation (12,500/hr App, 5,000/hr PAT, 900/2,000 pts/min, 100 concurrent secondary) so 30+ machines sharing one bucket trip 403/429 and most installs go stale — directly breaking §5.3; (b) Jira Cloud applies tenant-level cost throttling so N concurrent changelog backfills 429 the tenant; (c) the verdict cache (§9.3) is per-machine, so N developers each pay for byte-identical AI verdicts — WP-SPIKE-LLMCOST's 'per team per sync' denominator undercounts by the team-size factor; (d) §15 budgets a single 1× backfill. Also: no scheduler exists to run sync when Claude Code is closed (stdio server is torn down at session end; 'default to polling' names no poller), so the 'completeness guarantee' never runs unattended.

**Fix:** State the true per-installation rate semantics in §7.1 and that aggregate org load = per-install load × N; change WP-SPIKE-LLMCOST denominator to 'distinct AI subjects × active installs × per-verdict cost'; add a §15 NFR bounding aggregate API+LLM consumption at team scale. Either (a) make the §5.4 shared App-authenticated ingester + shared read-only snapshot + content-addressed shared verdict cache mandatory above a small team size (collapses ingestion/cost to 1× and fixes priority 5), or (b) model a per-machine budget = installation_limit / expected concurrent installers with jitter/lease + coordinated sync. Specify an actual sync trigger (Claude Code session-start hook AND/OR OS-scheduled job) and make tools loudly warn/refuse when watermark-lag exceeds a threshold.

**Target:** SPEC §7.1, §7.3, §5.3, §5.4, §9.3, §15; WORKPLAN WP-SPIKE-LLMCOST, WP-SYNC-ORCH, new WP-SPIKE-SYNCLOAD


### P7 · [CRITICAL] 'Ranking architecturally impossible' is false under the public default — any peer trivially scrapes per-person rows and sorts them; and public-by-default to unknown adopters almost certainly fails GDPR

**Problem:** N2/§17.5 market individual ranking as 'architecturally impossible (see §13)', but §11.1 persists per-person metric_snapshots readable by all installed members under the shipped default visibility=public, and §13.1 tools expose scope_type=person with no actor-identity/authorization gate — so any peer pulls all contributors' per-person rows and sorts client-side in seconds. §11.1 itself admits the guard governs only UI emission, not data access. WP-VISIBILITY's 'ranking-impossibility test' passes against the render path while the scrape vector stays open — a false sense of assurance. Separately, shipping peer-visible individual performance profiling as the out-of-the-box default (committed settings.json) to unknown OSS adopters defies the cited research and almost certainly fails GDPR legitimate-interest necessity/proportionality for any EU/UK employer; the only mitigation is a take-legal-advice disclaimer + an unenforced DPIA template (WP-LEGAL is a human flag with no gate). [Legal exposure flagged as scaffolding only — route the legal-basis determination through qualified data-protection counsel.]

**Fix:** Drop the 'architecturally impossible' language while public is reachable — it is factually false. Change the shipped default to self (matches research, minimizes blast radius); keep public as a friction-laden explicit opt-in that requires generating + acknowledging the DPIA/LIA first. Add a hard precondition in WP-VISIBILITY: scope_type=person snapshots cannot be written unless visibility=public AND a recorded LIA acknowledgement exists. Gate the per-person tool surface + flow-analyst behind explicit, logged authorization; compute scope_type=person on demand rather than persisting a bulk-readable per-person table.

**Target:** SPEC §2.2 N2, §4 D2, §11.1, §11.2, §17.5, §18 R1; WORKPLAN WP-VISIBILITY, WP-LEGAL, WP-GDPR-SCAFFOLD


### P8 · [HIGH] Division-by-zero / empty-window unhandled across ~8 ratio metrics — NaN/Infinity surfaced on ordinary inputs

**Problem:** CFR, Deployment Rework Rate, Flow Efficiency, Review Coverage, Say/Do, Predictability (1−CV with CV=stdev/mean), and Estimation bias are all bare ratios with no zero-denominator guard. A holiday week (0 deploys/0 incidents → CFR 0/0 → NaN), a new sprint (unset story points → Say/Do 0/0), n=1 sprint (CV div-0 → Infinity), or a binary-only PR (Review Coverage 0/0) produces NaN/Infinity in structuredContent, which either fails the numeric outputSchema (tool errors) or gets narrated as 'NaN%'. §8.6 specifies same-input⇒same-output but nothing about defined output for degenerate input; there is no engine-wide null/NaN convention. Triggers on the MOST ordinary day-one inputs, not exotic edges.

**Fix:** Add an engine-wide rule to §8.6: every ratio returns null (rendered 'N/A', data_quality='no_data') when the denominator is 0 — never NaN/Infinity. CV/predictability require n≥2 and mean>0 else null; estimation bias excludes 0-point estimates. Make outputSchema nullable and forbid the subagent from narrating null as a number. Add golden tests for each ratio metric on empty-window and zero-denominator inputs.

**Target:** SPEC §8.1, §8.2, §8.3, §8.5, §8.6, §13.1; WORKPLAN per-metric WPs


### P9 · [HIGH] Percentile estimator unspecified, no minimum-sample floors, and Monte Carlo determinism contract under-specified — all contradict the §5.3/§8.6 reproducibility promise

**Problem:** §8 mandates median+p75/p85/p90/p95 on every aggregation and §8.6 pins complexity counting/churn/DORA bands but conspicuously NOT the percentile algorithm — so two installs using simple-statistics (nearest-rank) vs d3.quantile (R-7) report different p75/p90 for identical synced data, directly contradicting §5.3. No metric has an n-floor before emitting p90/p95 (n=4 → p95 = single worst outlier), and §9.2.3's |z|>2 anomaly detector leaves 'minimum sample size' undefined. Separately, the Monte Carlo forecast claims 'deterministic given a seed' but NFR §15 forbids RNG without naming a specific cross-platform PRNG, sample iteration order, or float reduction — so two installs on different Node/arch can forecast different commit dates from identical throughput history.

**Fix:** Pin ONE percentile method (recommend type-7/linear or nearest-rank, documented) in §8.6 + formulaDoc. Add minimum-sample guards (n≥20 for p90, n≥30 for p95; below floor emit data_quality='insufficient_sample'); give the anomaly detector a numeric minimum-n. Vendor a pure-TS seeded PRNG (xorshift128+/PCG) in the engine version, make 'no Math.random in metric paths' a lint rule, fix sample iteration to a sorted canonical order, document forecasts are reproducible only per engine-version+seed. Add cross-implementation/cross-Node golden tests asserting identical percentiles and p50/p85/p95.

**Target:** SPEC §8 intro, §8.1, §8.2, §8.6, §9.2.3, NFR §15; WORKPLAN WP-METRICS-FLOW, WP-AI-ANOMALY, WP-PERF


### P10 · [HIGH] Opus 4.x rejects temperature/top_p/top_k (HTTP 400) but the harness hardcodes temperature:0 for every call — the D8 ensemble and flow-analyst subagent cannot run

**Problem:** D3/§9.1/§5.5/WP-AI-HARNESS specify 'temperature 0' as a blanket per-call rule and §6.2 persists a temperature column for every verdict, but the Anthropic API skill states verbatim that Opus 4.7/4.8 REMOVE sampling parameters and 400 on send. D8 escalates low-confidence verdicts to claude-opus-4-8 and §13.4 runs the flow-analyst subagent on claude-opus-4-8 — both through the same harness contract — so they fail with a 400 exactly when invoked. There is no per-model request-shape adapter anywhere. (Note: the determinism rationale for temp 0 is itself unsound — Anthropic states temp 0 never guaranteed identical outputs; pair this fix with reframing AI verdicts as point-in-time, not bit-reproducible.)

**Fix:** Add a per-model request-shape adapter: never send temperature/top_p/top_k/budget_tokens to a model id matching claude-opus-*. On Opus calls use adaptive thinking + low effort for determinism-seeking; store 'n/a (adaptive)' for Opus verdicts. Restate D3/§9.1 as 'temp 0 on Sonnet-tier only; Opus calls use adaptive thinking, no sampling params.' Add a harness test asserting no sampling param is ever sent to an Opus model. Separately add a model-deprecation runbook (retired ids 404) and state in §9/§10 that AI verdicts are point-in-time, not bit-reproducible; scope §8.6 'reproducible' to the deterministic engine only.

**Target:** SPEC §4 D3, §9.1, §9.3, §6.2, §13.4, §1/§8.6/§17 AC7 (reproducibility language); WORKPLAN WP-AI-HARNESS, WP-AI-CALIBRATION


### P11 · [HIGH] Reopen/re-Done has no dedupe/anchor policy, clock-skew unguarded, and squash/rebase-aware Lead Time mapping undefined

**Problem:** §8 formulas reference 'transition into Done' as if monotonic, but RESEARCH 924/1157/1161 prescribe 'first Done' and warn reopened issues produce multiple Done transitions; §8 never publishes a dedupe/anchor rule, so Recovery Time (first-resolve 1h vs last-resolve 25h), Throughput, Flow Time, Velocity and Time-in-Status are undefined for reopened items, and the 'now − t_k' open-segment term can go negative on future-dated transitions. Separately, §8.1 Lead Time SHA-joins on the on-branch authored date and is undefined for multi-PR deploys and rebase resets; the schema's pull_requests.first_commit_at (correct anchor) is not referenced and the §10 anti-gaming squash/rebase flag is not wired into the formula.

**Fix:** Publish a per-metric reopen policy in formulaDoc: Throughput/Velocity count once per window on FIRST Done and dedupe per issue per window; Flow Time stops at first Done with reopens as a separate rework counter-metric; Recovery Time = created→first-resolve plus a reopen-rate quality metric. Add clock-skew guards (clamp now−t_k≥0; flag future-dated transitions). For Lead Time: define the commit set via compare-API enumeration between consecutive deploy SHAs, compute per-commit and report median, anchor on pull_requests.first_commit_at, detect merge strategy and wire the §10 squash/rebase flag.

**Target:** SPEC §8.1, §8.2, §8.5, §8.6, §10; WORKPLAN WP-METRICS-DORA (add WP-JIRA-CHANGELOG dep), WP-METRICS-FLOW, WP-METRICS-AGILE


### P12 · [HIGH] HALOC has no file-classification layer — binaries vanish, generated/lockfiles inflate, formatter runs read as huge rework, rename detection unpinned — feeding four metrics under a per-person default

**Problem:** C2/§8.4 defines HALOC = Σ_hunk max(insertions,deletions) as the universal unit feeding churn%, work-type, PR size and Impact, with no binary/generated/rename/whitespace handling and §8.6 not pinning git diff/rename settings. A formatter run (30k reflowed lines) spikes a person's rework% and Impact; a 2MB binary swap is invisible (HALOC=0); a 50k-line regenerated OpenAPI client drowns everything; rename handling depends on unpinned git heuristics. Under the D2 per-person public default these land on individuals — §6.3 itself warns 'wrong stitches produce defamatory stats'. WP-CODE-ANALYSIS edge-case tests cover pure-add/delete/modify but NOT binary/generated/rename/whitespace.

**Fix:** Define HALOC over a normalized diff in formulaDoc: pin git rename-detection threshold (recorded in engine version); classify and bucket-or-exclude binary + generated/vendored paths (gitattributes linguist-generated + configurable globs like *-lock.json, dist/**); offer a whitespace-insensitive mode (git diff -w); count rename-with-edits as edit hunks only; surface excluded volume as a separate 'generated/binary' figure rather than silently zeroing. Add binary/generated/rename/whitespace cases to the WP-CODE-ANALYSIS edge-case suite.

**Target:** SPEC §1 C2, §8.4, §8.6, §10; WORKPLAN WP-CODE-ANALYSIS


### P13 · [HIGH] Identity grain: squash co-author loss, 0.8 local-part+name auto-merge of distinct humans, no schema-level is_bot, email-anchored identity fragments on email change

**Problem:** Four related identity defects, all distorting the opt-in per-person view and (where humans straddle teams) aggregates: (1) commits is single-author with no co-author join table or trailer parsing, so squash-merged pair work loses co-authors; (2) §6.3 auto-merges at the 0.8 'local-part+name' tier, so john.smith@acme.com and john.smith@vendor.com auto-merge into one person (RESEARCH 1379 warns of this); (3) bots are handled only as un-matchable noreply emails — no is_bot attribute — so dependabot/renovate/ci accounts become 'developers' and can top a per-person view, and ~15 metrics that require 'non-bot' filtering re-detect ad hoc and disagree; (4) persons are anchored on primary_email_hash so a rebrand/email change fragments one human into thirds. Merges are reversible/audited and WP-SPIKE-IDENTITY/R4 gates per-person ship, which keeps these high not critical.

**Fix:** Add commit_authors(commit_sha, identity_id, role, source) + parse Co-authored-by:/Signed-off-by: trailers; prefer pre-squash PR commits for squash merges. Demote the 0.8 local-part+name tier to the human-confirm queue (never auto), auto-merge only on verified full-email/GitHub-verified email↔login, add per-org domain allowlist + split-detector. Add identities.is_bot (GitHub type==Bot, [bot] suffix, App author, configurable allowlist) + person-creation guard + default exclusion, routed through the existing WP-METRICS-PR bot-filtering requirement so it is schema-level. Anchor person identity on stable account ids (GitHub user id, Jira accountId), email as an attribute. Make WP-SPIKE-IDENTITY explicitly validate-or-demote the 0.8 gate.

**Target:** SPEC §6.1 (commits), §6.3; WORKPLAN WP-SPIKE-IDENTITY, WP-IDENTITY, WP-METRICS-PR, WP-DB-SCHEMA


### P14 · [HIGH] Sprint membership boolean, no issue hierarchy (subtask double-count), and unprovenanced story-point scalar corrupt velocity/Say-Do

**Problem:** Three agile-data-model gaps that produce wrong velocity/Say-Do on ordinary inputs: (1) sprint_issues is a single boolean (added_after_start) that cannot represent removal, add-then-remove, or mid-sprint re-point — RESEARCH 926 needs the sprint-field changelog; (2) issues has type but no parent_id/epic_link/is_subtask, so pointed subtasks sharing a sprint with a pointed parent double-count velocity (RESEARCH 923 mandates subtask dedup; 732 needs hierarchy for Investment Balance down-propagation) — the parent field exists only in raw JSON, never normalized; (3) issues.story_points is a bare scalar with no field-id provenance, but §7.2 discovers the field per-instance while dual-field multi-project instances (customfield_10016 vs _10026) leave half the org NULL with no column to diagnose the 0-velocity artifact.

**Fix:** Replace sprint_issues with sprint_membership_events(sprint_id, issue_id, change ENUM(added|removed), points_at_event, transitioned_at, was_present_at_start) derived from the sprint-field changelog. Add parent_id/epic_key/is_subtask + issue_hierarchy_level to issues, normalize from raw, count velocity at one configurable hierarchy level, roll subtask points to parent, propagate Investment category down. Store story_points_field_id + story_points_raw, resolve per-PROJECT, validate numeric (else null+flag), surface per-project estimation-field-coverage. Add subtask-dedup + scope-removal + dual-field golden fixtures to WP-METRICS-AGILE.

### Ingestion & sync correctness (GitHub + Jira)

_Verdict: The attacking review is substantially correct and survives refutation: 7 of 11 findings are upheld at the claimed severity (3 critical, 4 high) and the remaining 4 are downgraded to medium but remain real, leaving zero overturned. The dimension is not fit for purpose as written. The root cause is confirmed by the source documents, not just inferred: RESEARCH.md designed the ingestion/sync model for a SINGLE server-side ingester — it tags every layer '(server-side)' (lines 200-214), prescribes a 'token-per-installation pool' routing each org to its own bucket (1092), and explicitly recommends 'run the MCP server as a remote Streamable HTTP backend (single ingestion path, one shared DB) rather than N local stdio servers that drift' (1466). SPEC D1 is a [user] decision that inverted this to local-first per-user, then carried the server-side rate-limit and sync numbers across §7 without re-deriving the consequences. That inversion breaks three load-bearing claims at once: (1) the per-installation rate budget is now shared and divided across every developer's machine, so concurrent re-syncs trip 403/429 and leave most installs stale (the attack's '80 req/min' figure is wrong — it is 900/2,000 pts/min + 100 concurrent — but the divide-not-multiply thesis stands); (2) the 'completeness guarantee' has no scheduler because the stdio MCP server only exists while Claude Code is open and §7.3 'default to polling' names no poller; (3) the §5.3 'identical data' transparency thesis is false under per-user Jira 3LO (and PAT-fallback GitHub) authorization, with no completeness flag to warn anyone — though the org-installed GitHub App path partially mitigates the GitHub half. Beyond the topology, the correctness core has genuine, independently-valid gaps: upsert-only sync with no tombstoning (deleted issues/PRs and force-pushed commits double-counted forever), the Jira bulk-changelog 'fix' relocates rather than removes the 100-history truncation because the bulk endpoint is itself paginated, and large-PR GraphQL hydration drops inner-connection data the research had already told them to paginate. The downgraded items (SQLite responsiveness, ETag-on-list-endpoints, OAuth rotation, crash-mid-page child hydration, updated_at tie-breaks) are real and worth fixing but are config/robustness-level or narrowed by the default poll-only/own-file local topology. Recommendation: treat the three criticals as gating — either make the §5.4 shared-ingester path mandatory for teams, or fully re-derive §7 for the per-user local-first topology (per-machine budget + coordination, an explicit scheduler, and per-credential completeness flags) — and schedule the tombstoning, changelog-exhaustion, and inner-GraphQL-pagination highs into WP-GH-SYNC / WP-JIRA-CHANGELOG / WP-GH-CLIENT acceptance with the missing golden tests._

### Security, privacy & GDPR-legal contradiction

_Verdict: Not fit for purpose on this dimension. After attempting to refute each finding, the two critical findings survive intact and are genuinely disqualifying: (1) the flagship ethical guarantee that individual ranking is 'architecturally impossible' (N2, §17.5) is contradicted by the spec's own §11.1, which persists per-person metric_snapshots readable by all peers under the shipped public default and explicitly admits the anti-ranking guard governs only rendering, not data access — so any viewer assembles a forced ranking by querying scope_type=person rows; (2) shipping peer-visible individual performance profiling as the out-of-the-box default to unknown OSS adopters directly defies the cited research and almost certainly fails GDPR legitimate-interest balancing for any EU/UK employer, with the only mitigation being a take-legal-advice disclaimer and an unenforced template (WP-LEGAL is a human flag with no gate). Two further high findings hold: 'hashed at rest' presents a trivially reversible unsalted hash of a low-entropy corporate email as the privacy-at-rest control, and 'raw payloads scrubbed of tokens' is asserted in the NFRs with no mechanism, no owning work item, and no DoD test, while the raw-retention design simultaneously reintroduces the cleartext emails the hash was meant to minimize. The 'per-tenant isolation' claim is medium — vacuous and unenforceable local-first, masking a real cross-client-mixing risk. Two findings were tempered on refutation: the env-var secret-exposure finding is downgraded to low (real hardening gap but follows the only sanctioned plugin secret-delivery convention and is a same-UID-local threat, not a spec-level contradiction), and the parameterized-SQL finding stays low (no enforcement gate is a real gap, but the cited injection vector is largely defused by the §7.2 numeric-status-ID mandate). The missingItems are all re-validated as genuinely absent — most importantly at-rest DB encryption, a threat model, an enforced DPIA/LIA precondition, access control on the per-person tool surface, supply-chain integrity for the WASM grammars, and erasure completeness against raw/ai_verdicts residue. The fixes are tractable (flip the default to self with public as a gated opt-in, build a real scrubber, encrypt the DB, stop calling hashing a privacy control, harden secret handling, add supply-chain integrity), but as written this dimension ships a GDPR exposure as the default and a self-contradicting privacy promise as the headline. Legal exposure here is flagged as scaffolding only and must be routed through qualified data-protection counsel._

### Work plan executability: dependencies, circularity, sizing, DoD

_Verdict: The attack is directionally correct: the WORKPLAN is well-written prose but is not yet safely fan-out-executable as written, and several first-class capabilities are specced-and-depended-upon but unbuilt. After adversarial refutation, the findings stratify rather than uniformly hold. Two genuine hard defects survive at critical/high: (1) WP-METRICS-FLOW (Wave 4) depends on the per-workflow Flow State Model (C3), which is AI-seeded in Wave 5 AND has no producing work item at all — a backward edge plus a missing WP that, left as-is, forces the exact silent-false-dashboard the product exists to prevent (critical, upheld); and (2) the Wave-0 monolithic golden-expectations/dataset contradicts both the per-metric goldenFixtures already in the §8.6 contract and the fact that several metric formulas aren't pinned until Waves 1/4/5, making DoD item 4 partially non-executable and risking shared-mutable-fixture collisions under parallelism (high, downgraded from two critical findings). The remaining findings are real but overstated and downgrade to medium/low: the no-sizing claim is a scheduling-optimization gap, not a correctness blocker (medium); the WP-IDENTITY/§12 contradiction is a deletable prose inconsistency given the authoritative Depends-on edges (medium); the 'adversarial review has no agent' finding largely fails because the gate is harness-provided process review like WP-LEGAL, leaving only a missing rubric (low); the AI gold-set 'circular dependency' is refuted (fixture gold sets are distinct from the per-team calibration protocol), leaving only unpinned numeric targets (low); the plugin-install e2e claim is mostly refuted by WP-MCP-SERVER's boot+handshake test (low); and the WP-LEGAL gating claim is mostly refuted because the SPEC MUST binds the EU/UK-deployment claim, not the OSS publish (low). Genuine missing/underspecified items worth fixing before fan-out: the Flow State Model WP, re-derivation+version-stamping (incl. an engine_version column on metric_snapshots), team/org topology+rollup, payload-versioning for safe re-derivation, per-metric golden-fixture colocation, pinned AI numeric targets, and long-pole sizing tags.   - Breaking example: repos:['ORG/payments'] is renamed to 'ORG/billing'; node_id is unchanged but the reconciliation sweep queries the configured 'ORG/payments' and gets 404, so the cursor never advances and sync silently stalls. Separately, adding 'ORG/app-fork' ingests upstream commits/PRs and inflates the team's deployment frequency and lead time with work they never did.
  - Fix: Resolve configured owner/name to node_id once and track renames via node_id; treat 404-on-known-node_id as a rename/transfer and re-resolve; store archived/fork flags on repositories and exclude forks from human-work aggregates by default; surface 'repo no longer reachable' as a doctor/sync error.
  - Reasoning: Refutation failed. SPEC keys repositories and idempotency on github_node_id (§6.1, §7.4) — good for storage stability — but config is name-based ('repos: [ORG/app]', §12.4/§14) and nothing reconciles the configured owner/name against the stable node_id. There is no archived/fork flag on repositories, no rename/transfer detection, and no fork exclusion from human-work aggregates. The reconciliation sweep (§7.3) is the completeness guarantee but it polls by the configured (stale) name, so a rename yields a silent 404 and a never-advancing cursor. Forks would have their upstream-merged commits/PRs counted in throughput/DORA. Both breaking examples are plausible and unaddressed. Medium severity: real and likely to bite real installs, but it degrades specific scopes rather than corrupting the core engine universally, and the node_id-keyed storage limits blast radius.

- **[low/downgraded] No export/reporting surface for the leader personas who must share results outside the IDE**
  - Location: SPEC §3, §13, §17; WORKPLAN Wave 8
  - Breaking example: An eng org leader needs a quarterly DORA + investment-balance board deck and can only get conversational/structured output in a Claude Code session — no get_export tool, no per-team CSV of the four keys, no PDF — so they fall back to screenshots.
  - Fix: Add an export MCP tool (structured CSV/JSON for any metric bundle + scope + window) and an optional report-generation path (markdown/PDF board pack from snapshots); add WP-EXPORT to Wave 7/8; consider scheduled/periodic reports for the leader personas.
  - Reasoning: Partly refuted. The personas (Eng org leader, Delivery/agile lead) plausibly want shareable artifacts, and the product surface is indeed in-session MCP tools plus @-mentionable dashboard Resources (§13.2) with no get_export tool, no CSV/PDF, no scheduled report. However, the product is DELIBERATELY scoped (G4, §5.1) as a thin in-IDE Claude Code plugin, and tool outputs are structured/machine-checkable (outputSchema+structuredContent), so a minimal CSV/JSON export is a trivial additive layer rather than a missing foundational capability, and a user can already have the model emit structured output. This is a legitimate scope-completeness observation for the leader personas but not a correctness break and not foundational — downgraded from medium to low.

- **[low/downgraded] Partial-sync failures and AI refusals are stored but not propagated into per-metric data_quality**
  - Location: SPEC §6.2 sync_state, §15 Observability, §7.5
  - Breaking example: GitHub sync of repo A succeeds but repo B's deployments fetch hits the secondary rate limit and aborts; sync_state.error for B is set, but get_dora returns a deployment-frequency number over A only with data_quality not necessarily reflecting B's missing deploys, so the figure looks complete unless the user separately calls sync_status.
  - Fix: Propagate per-resource sync errors and AI refusals into the data_quality field of every affected metric/verdict output (e.g. 'partial: repo B deploys missing'); make tools refuse to present a metric as complete when a contributing resource's sync failed.
  - Reasoning: Partly refuted. The spec does provide more surfacing than the finding allows: §7.5 surfaces Sync Freshness/watermark-lag 'on every report header' and as a tool, every metric/verdict output carries a data_quality field (§6.2/§13.1), and doctor checks sync freshness. The valid residual gap is narrow: the spec does not specify WIRING per-resource sync_state.error or AI refusals INTO the data_quality of the specific affected metrics, so a DF number over a partially-failed multi-repo sync could present as complete while the failure shows only in a separate sync_status call. The building blocks all exist; only the propagation rule is unspecified. Downgraded from medium to low — a refinement of an existing trust-signal surface, not a missing capability.
